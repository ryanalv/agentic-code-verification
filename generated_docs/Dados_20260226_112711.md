# Documentação Técnica Oficial – Pipeline de Feitos (`feitos.py`)

---

## 1. Visão Geral

O arquivo `feitos.py` implementa um pipeline de ingestão, transformação e persistência de dados judiciais, focado na entidade "feitos". Ele opera sobre dados extraídos de uma coleção MongoDB, realiza transformação rigorosa para um schema tabular, e persiste o resultado em um Data Lake baseado em Apache Iceberg, usando o formato Parquet e compressão zstd. O objetivo é garantir dados limpos, normalizados e analiticamente otimizados, prontos para consumo por sistemas analíticos, BI e machine learning.

---

## 2. Contexto Teórico da Arquitetura

### 2.1. Tecnologias e Padrões Utilizados

#### **Apache Spark (PySpark)**
- **O que é:** Framework de processamento distribuído, capaz de manipular grandes volumes de dados em clusters.
- **Por que usar:** Escalabilidade, paralelismo e tolerância a falhas, essenciais para processar dados judiciais volumosos.

#### **MongoDB**
- **O que é:** Banco NoSQL orientado a documentos, com dados semi-estruturados e potencialmente aninhados.
- **Por que usar:** Flexibilidade para armazenar registros judiciais complexos.

#### **Apache Iceberg**
- **O que é:** Formato de tabela para Data Lake, com versionamento, particionamento eficiente e suporte a operações ACID.
- **Por que usar:** Permite consultas eficientes, histórico de versões e integração com ferramentas analíticas.

#### **Parquet**
- **O que é:** Formato colunar otimizado para compressão e leitura analítica.
- **Por que usar:** Reduz custos de armazenamento e acelera queries.

#### **Zstandard (zstd)**
- **O que é:** Algoritmo de compressão moderno, balanceando velocidade e taxa de compressão.
- **Por que usar:** Minimiza uso de disco sem sacrificar performance.

#### **Arquitetura Modular & Pipeline Determinístico**
- **O que é:** Separação clara entre etapas (extração, transformação, persistência), com contratos explícitos de entrada e saída.
- **Por que usar:** Facilita manutenção, evolução e integração com orquestradores.

## 4. Detalhamento Técnico

### 4.1. Definição de Schemas

#### **Schema de Origem (MongoDB)**
- Definido explicitamente via `StructType` e `StructField`.
- Tipos: StringType, IntegerType, TimestampType, LongType, BooleanType, etc.
- Campos aninhados são mapeados para subcampos (ex: `classe.nome`).

#### **Schema de Destino (Iceberg)**
- Criado via DDL em `query_create_table`.
- Colunas flat, nomes padronizados.
- Particionamento por data (`job_dh_ref`).
- Compressão Parquet/zstd.

---

### 4.2. Função de Transformação `_transform`

#### **Assinatura**
```python
def _transform(df: DataFrame) -> DataFrame
```
- **Entrada:** DataFrame Spark com schema MongoDB.
- **Saída:** DataFrame Spark tabular, normalizado.

#### **Fluxo de Execução**

1. **Seleção de Campos**
    - Utiliza `df.select()` para mais de 150 campos.
    - Inclui campos simples, aninhados e de controle.

2. **Renomeação (Aliases)**
    - Todos os campos recebem aliases explícitos.
    - Evita ambiguidades e padroniza nomenclatura.

3. **Desaninhamento**
    - Campos aninhados são extraídos (ex: `classe.nome` → `classe_nome`).

4. **Casting de Tipos**
    - Campos como `versao` são convertidos explicitamente para int.

5. **Retorno**
    - DataFrame resultante pronto para persistência.

#### **Pseudocódigo**
```
Para cada campo no schema de destino:
    Se campo é simples:
        selecione e renomeie
    Se campo é aninhado:
        selecione subcampo e renomeie
    Se campo exige casting:
        aplique casting
Retorne DataFrame final
```

#### **Casos de Erro**
- Campo ausente: coluna nula.
- Tipo incompatível: falha de execução.
- Nome incorreto: pipeline interrompido.
- Estrutura aninhada inesperada: campo nulo ou erro.

---

### 4.3. Objeto de Configuração `transformer_config`

#### **Campos Principais**
- **job_name:** Nome do job.
- **spark_master, spark_executor_cores, etc.:** Recursos Spark.
- **mongodb_database_name, mongodb_collection_name:** Fonte dos dados.
- **iceberg_catalog_name, iceberg_namespace, iceberg_table_name:** Destino.
- **mongodb_schema:** Schema de entrada.
- **query_create_table:** DDL de tabela Iceberg.
- **filter_columns:** Colunas de data para filtragem incremental.
- **refresh_rate, refresh_rate_history, first_run_date:** Janela temporal.
- **transform:** Função de transformação.

#### **Integração**
- Orquestradores (ex: Airflow) usam o config para parametrizar execução.
- DataFrame lido é passado para `_transform`.
- DataFrame final é persistido conforme DDL.

---

### 4.4. Fluxo Detalhado

1. **Leitura MongoDB**
    - Extração da coleção `feitos` usando schema explícito.

2. **Transformação**
    - `_transform` aplica seleção, renomeação, casting.

3. **Filtragem Incremental**
    - Usa `filter_columns` e parâmetros de data.

4. **Persistência Iceberg**
    - Tabela criada conforme DDL, particionada por dias, compressão zstd.

5. **Consumo**
    - Dados disponíveis para BI, ML, relatórios.

## 6. Contratos Públicos

- **Função `_transform(df: DataFrame) -> DataFrame`:**  
  Contrato de transformação determinística, recebe DataFrame MongoDB, retorna DataFrame tabular.

- **Objeto `transformer_config`:**  
  Contrato de configuração, expõe todos os parâmetros necessários para integração e execução.

- **Schema `mongodb_schema`:**  
  Contrato formal do formato de entrada, define campos, tipos, estruturas.

- **Query `query_create_table`:**  
  Contrato formal do formato de saída, tipos, nomes, particionamento, propriedades.

- **Campos `filter_columns`:**  
  Define colunas de data para filtragem incremental.

- **Parâmetros de atualização:**  
  Controlam janela temporal de processamento.

## 7. Riscos, Limitações e Tradeoffs

### **Riscos**
- Mudanças no schema de origem podem quebrar o pipeline.
- Dados inconsistentes causam falhas ou perda de informação.
- Crescimento de volume pode impactar performance.
- Dependência de nomes exatos exige manutenção manual.

### **Limitações**
- Pipeline rígido, não tolera campos inesperados.
- Sem tratamento de erro customizado.
- Sem enriquecimento dinâmico ou lógica condicional.
- Campos complexos ignorados ou convertidos em string.

### **Tradeoffs**
- Schema flat facilita análise, mas perde contexto relacional.
- PySpark traz escalabilidade, mas aumenta complexidade operacional.
- Compressão zstd reduz espaço, mas consome mais CPU.
- Particionamento por data acelera queries, mas pode gerar fragmentação.
- Aliases explícitos aumentam clareza, mas exigem manutenção manual.
