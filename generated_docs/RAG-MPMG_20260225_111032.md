 # DocumentaĂ§ĂŁo TĂŠcnica Exaustiva: Sistema RAG-MPMG

## 1. VisĂŁo Geral Executiva

O **RAG-MPMG** ĂŠ uma plataforma de InteligĂŞncia Artificial JurĂ­dica desenvolvida para o MinistĂŠrio PĂşblico de Minas Gerais (MPMG), implementando o padrĂŁo arquitetural **RAG (Retrieval-Augmented Generation)** para processamento, anĂĄlise e geraĂ§ĂŁo de documentos jurĂ­dicos. 

Em termos acessĂ­veis: imagine um assistente jurĂ­dico extremamente eficiente que, antes de responder qualquer pergunta ou redigir uma minuta, consulta milhares de documentos processuais (processos, denĂşncias, arquivamentos) para basear suas respostas em fatos reais e legislaĂ§ĂŁo aplicĂĄvel. Este sistema ĂŠ o "cĂŠrebro" digital que permite que promotores de justiĂ§a gerem minutas jurĂ­dicas, analisem documentos processuais e conversem com seus processos atravĂŠs de linguagem natural.

### 1.1. PropĂłsito e Escopo

O sistema atua em trĂŞs frentes principais:
1. **GeraĂ§ĂŁo Inteligente de Minutas**: CriaĂ§ĂŁo automĂĄtica de denĂşncias, arquivamentos e representaĂ§Ăľes baseadas em documentos processuais existentes
2. **Processamento OCR em Lote**: ExtraĂ§ĂŁo de texto de documentos PDF, HTML e DOCX provenientes de diferentes fontes (SRU - Sistema de Registro Ănico, MPE - MinistĂŠrio PĂşblico EletrĂ´nico, e documentos eletrĂ´nicos diversos)
3. **ModificaĂ§ĂŁo Contextual de Documentos**: RevisĂŁo e adaptaĂ§ĂŁo de minutas existentes com base em instruĂ§Ăľes especĂ­ficas do usuĂĄrio, mantendo o contexto jurĂ­dico original

### 1.2. CaracterĂ­sticas Arquiteturais Distintivas

- **ResiliĂŞncia Multi-Cloud**: Implementa mecanismo de fallback automĂĄtico entre Azure OpenAI e Google Gemini, com controle de concorrĂŞncia via locks distribuĂ­dos em Redis
- **Processamento DistribuĂ­do**: Utiliza Ray para paralelizaĂ§ĂŁo de tarefas de OCR em mĂşltiplos nĂłs computacionais
- **Arquitetura HĂ­brida de PersistĂŞncia**: Combina MongoDB (documentos estruturados), SQL Server (dados processuais legados) e HDFS (arquivos binĂĄrios em escala)
- **GestĂŁo de Filas PrioritĂĄrias**: Sistema de enfileiramento em Redis com trĂŞs nĂ­veis de prioridade (high/default/low) e rastreamento completo do ciclo de vida dos jobs

## 2. Contexto TeĂłrico da Arquitetura

Antes de mergulharmos na implementaĂ§ĂŁo especĂ­fica, ĂŠ fundamental compreender os padrĂľes arquiteturais e tecnologias que fundamentam este sistema. Esta seĂ§ĂŁo estabelece o vocabulĂĄrio tĂŠcnico necessĂĄrio para entender as decisĂľes de design.

### 2.1. RAG (Retrieval-Augmented Generation)

**Conceito TeĂłrico**: RAG ĂŠ uma arquitetura de IA que combina recuperaĂ§ĂŁo de informaĂ§Ăľes (Information Retrieval) com geraĂ§ĂŁo de linguagem natural. Em vez de um modelo de linguagem (LLM) responder baseado apenas em seu conhecimento prĂŠvio treinado, o sistema primeiro "busca" documentos relevantes em uma base de conhecimento e depois gera a resposta fundamentada nesses documentos.

**AplicaĂ§ĂŁo no CĂłdigo**: 
- A classe `VectorStoreService` gerencia a indexaĂ§ĂŁo de textos OCR em vector stores (bancos vetoriais) da OpenAI
- O `PromptBuilder` (em `inteligencia_schema.py`) constrĂłi contextos ricos concatenando textos de documentos recuperados antes de enviar ao LLM
- O fluxo `/gerar-minuta` exemplifica o padrĂŁo: recupera documentos do processo â extrai texto via OCR â indexa em vector store (opcional) â constrĂłi prompt com contexto â gera minuta

### 2.2. Vector Stores e Embeddings

**Conceito TeĂłrico**: Um vector store armazena representaĂ§Ăľes matemĂĄticas (embeddings) de textos em espaĂ§os multidimensionais (tipicamente 1536 dimensĂľes). Textos semanticamente similares ficam prĂłximos neste espaĂ§o vetorial, permitindo busca por similaridade (similarity search) alĂŠm da busca por palavras-chave exatas.

**AplicaĂ§ĂŁo no CĂłdigo**:
- `OpenAIClient` gerencia vector stores via API compatĂ­vel com OpenAI (`/v1/vector_stores`)
- FunĂ§ĂŁo `insert_texto_to_vector_store` converte texto em arquivo .txt e associa a um vector store para consulta semĂ˘ntica futura
- `remove_file_expired` implementa garbage collection de arquivos expirados nos vector stores, mantendo a higiene da base vetorial

### 2.3. Arquitetura em Camadas (Layered Architecture)

**Conceito TeĂłrico**: SeparaĂ§ĂŁo de responsabilidades em camadas horizontais: API (interface), Casos de Uso (orquestraĂ§ĂŁo), ServiĂ§os (lĂłgica de domĂ­nio), RepositĂłrios (acesso a dados) e Infraestrutura (conexĂľes externas). Cada camada sĂł pode conversar com a camada imediatamente inferior.

**AplicaĂ§ĂŁo no CĂłdigo**:
- **Camada de API** (`app/api/v1.py`): Apenas recebe HTTP requests e delega para filas
- **Camada de Casos de Uso** (`app/usecases/`): Orquestra mĂşltiplos serviĂ§os (OCR, persistĂŞncia, validaĂ§ĂŁo)
- **Camada de ServiĂ§os** (`app/services/`): ContĂŠm lĂłgica pura de negĂłcio (GPT, OCR)
- **Camada de RepositĂłrios** (`app/repositories/`): Abstrai acesso a MongoDB, SQL Server, HDFS
- **Camada de Infraestrutura** (`app/core/`): ConexĂľes com Redis, Mongo, configuraĂ§Ăľes

### 2.4. Sistema de Filas e Processamento AssĂ­ncrono

**Conceito TeĂłrico**: Em sistemas de IA, operaĂ§Ăľes como OCR em PDFs pesados ou geraĂ§ĂŁo de texto via LLM sĂŁo demoradas (10s a minutos). Um sistema sĂ­ncrono (HTTP request-response) bloquearia o cliente. Filas de mensagens (message queues) desacoplam a requisiĂ§ĂŁo do processamento, permitindo execuĂ§ĂŁo assĂ­ncrona e resiliente.

**AplicaĂ§ĂŁo no CĂłdigo**:
- `QueueManager` implementa filas em Redis com mĂşltiplas prioridades (high/default/low)
- Jobs sĂŁo enfileirados via `enqueue_task` e consumidos por workers Ray via `dequeue_task`
- Status tracking completo: pending â processing â completed, com TTL (Time-To-Live) de 24h

### 2.5. OCR (Optical Character Recognition) e Processamento de Documentos

**Conceito TeĂłrico**: OCR ĂŠ a tecnologia que converte imagens de texto (escaneados ou PDFs nativos) em texto editĂĄvel. Modelos modernos como Surya utilizam deep learning (redes neurais convolucionais + transformers) para reconhecimento de layout e texto.

**AplicaĂ§ĂŁo no CĂłdigo**:
- `ocr_service.py` encapsula Surya OCR para PDFs (conversĂŁo pĂĄgina-a-pĂĄgina em imagens 300 DPI)
- Suporte multi-formato: PyMuPDF (PDF), BeautifulSoup (HTML), python-docx (Word)
- Pipeline de validaĂ§ĂŁo: tolerĂ˘ncia mĂĄxima de 10% de falhas em lotes de OCR (`validar_ocrs`)

**Endpoints Implementados**:

4. **OCR de Documentos de Processo** (`POST /v1/ocr-documentos-processo`):

#### 4.2.3. Gerenciamento de Filas (`app/core/queue.py`)

ImplementaĂ§ĂŁo de **Message Queue** customizada sobre Redis, com suporte a mĂşltiplas prioridades e tracking de estado.

**Constantes e ConfiguraĂ§Ăľes**:
```python
QUEUE_PREFIX = "fila_ray"
JOB_STATUS_EXPIRATION = 86400  # 24 horas em segundos
STATUS_PENDING = "pending"
STATUS_PROCESSING = "processing"
STATUS_COMPLETED = "completed"
DEFAULT_QUEUE = "default"
HIGH_PRIORITY_QUEUE = "high"
LOW_PRIORITY_QUEUE = "low"
```

**Classe `QueueManager`**:

*MĂŠtodo `enqueue_task(data: Dict[str, Any], queue: str = DEFAULT_QUEUE) -> str`*:
1. Gera `job_id` UUID se nĂŁo fornecido no payload
2. Serializa `data` para JSON
3. Armazena metadados do job em Redis (hash) com chave `fila_ray:job:{job_id}:status`:
   - `status`: "pending"
   - `enqueue_at`: timestamp ISO
   - `queue`: nome da fila
   - `data_raw`: payload JSON original (para reprocessamento em caso de falha)
4. Empurra (RPUSH) o payload JSON na lista Redis `fila_ray:{queue}`
5. Retorna `job_id`

*MĂŠtodo `dequeue_task(queues=None, block=True, timeout=5) -> Tuple[str, Dict]`*:
1. Converte nomes de filas em chaves Redis completas (`fila_ray:{queue}`)
2. Se `block=True`: utiliza `BLPOP` (Blocking Left Pop) que aguarda atĂŠ `timeout` segundos por elementos em qualquer uma das filas listadas (ordem define prioridade)
3. Se `block=False`: itera filas com `LPOP` nĂŁo-bloqueante
4. Ao obter item:
   - Desserializa JSON
   - Atualiza status no Redis para "processing" com timestamp `processing_at`
   - Retorna tupla (nome_fila_original, dados_job)

*MĂŠtodo `_set_job_status(job_id, status, queue, data_raw=None)`*:
- Implementa upsert de status com TTL automĂĄtico (24h)
- Preserva campos histĂłricos (nĂŁo sobrescreve timestamps anteriores)
- Estrutura de dados versionada por status:
  - `pending`: inclui `enqueue_at`, `data_raw`
  - `processing`: inclui `processing_at`
  - `completed`: inclui `completed_at`

*MĂŠtodo `queue_stats(queue: str = None) -> Dict`*:
- Varre todas as chaves `fila_ray:job:*:status` no Redis (usando SCAN implĂ­cito)
- Agrega mĂŠtricas por fila:
  - Contadores: `pending`, `processing`, `completed`, `total`
  - Tempos: `avg_wait_time` (processing - enqueue), `avg_process_time` (completed - processing), `avg_time_elapsed` (completed - enqueue)
- Calcula tempos mĂĄximos e mĂŠdias
- Permite filtragem por fila especĂ­fica

*MĂŠtodo `requeue_processing_jobs() -> int`*:
- RecuperaĂ§ĂŁo de falhas (failover): varre jobs com status "processing"
- Re-enfileira jobs que possam ter sido perdidos devido a crash de workers
- Retorna contagem de jobs recuperados

#### 4.7.2. Caso de Uso de OCR (`app/usecases/ocr_usecase.py`)

## 6. ConsideraĂ§Ăľes de ResiliĂŞncia e Escalabilidade

### 6.1. Mecanismo de Circuit Breaker e Fallback

O sistema implementa um sofisticado mecanismo de **fallback** entre provedores de LLM:

1. **Tentativa PrimĂĄria**: Sempre tenta Azure OpenAI primeiro (melhor performance/latĂŞncia presumida)
2. **DetecĂ§ĂŁo de Falha**: Captura exceĂ§Ăľes especĂ­ficas (`RateLimitError`, `InternalServerError`, `ReadTimeout`, `BadRequestError`, `RemoteProtocolError`)
3. **Lock DistribuĂ­do**: Utiliza Redis com chave `lock:gemini:active` e TTL de 60 segundos para evitar que mĂşltiplas threads faĂ§am "thundering herd" para a API do Gemini simultaneamente apĂłs uma falha do Azure
4. **PersistĂŞncia de Estado**: Registra qual cliente foi efetivamente utilizado (`client_geracao`) no documento MongoDB para auditoria e debugging

### 6.2. Processamento DistribuĂ­do com Ray

- **ParalelizaĂ§ĂŁo**: Tarefas de OCR (CPU-intensive) sĂŁo distribuĂ­das via Ray para mĂşltiplos nĂłs/workers
- **Runtime Environment**: ConfiguraĂ§ĂŁo garante que workers remotos tenham acesso Ă s mesmas variĂĄveis de ambiente e dependĂŞncias do nĂł principal
- **Rastreamento**: `get_response_task_ray` extrai metadados (`task_id`, `job_id`) para correlaĂ§ĂŁo entre jobs Redis e tasks Ray

### 6.3. GestĂŁo de Filas PrioritĂĄrias

- **MĂşltiplas Filas**: SeparaĂ§ĂŁo fĂ­sica em Redis (`fila_ray:high`, `fila_ray:default`, `fila_ray:low`)
- **PriorizaĂ§ĂŁo**: Uso de `BLPOP` com ordem de filas como argumento garante que jobs "high" sejam consumidos antes de "default"
- **Reprocessamento**: `requeue_processing_jobs` permite recuperaĂ§ĂŁo automĂĄtica de jobs que falharam durante processamento (crash do worker)

### 6.4. TolerĂ˘ncia a Falhas em OCR

- **ValidaĂ§ĂŁo de Qualidade**: Rejeita lotes onde mais de 10% dos documentos falharam no OCR (prevenĂ§ĂŁo de "garbage in, garbage out")
- **Processamento Individual**: Falha em um documento nĂŁo aborta o lote inteiro (erro ĂŠ capturado em `processar_resposta_ray`, registrado em `warnings`, e o processamento continua para os demais)
