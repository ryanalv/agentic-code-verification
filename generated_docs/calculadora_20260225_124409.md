# 2. CONTEXTO TEÓRICO DA ARQUITETURA

Antes de prosseguirmos para a análise diagnóstica dos erros e modelo arquitetural proposto, faz-se necessário estabelecer o arcabouço teórico que fundamentaria a análise do projeto, caso os artefatos estivessem disponíveis. Esta seção atende ao requisito de documentação para público misto, estabelecendo conceitos desde fundamentos até aplicações práticas.

## 2.1. O Paradigma da Tríade Web: HTML, CSS e JavaScript

Para compreender uma calculadora web, devemos entender a separação de responsabilidades tripartite que governa o desenvolvimento front-end moderno:

**HTML (HyperText Markup Language) - A Estrutura Óssea:**
- **Conceito Teórico:** Linguagem de marcação semântica que define a hierarquia e o significado estrutural do conteúdo. Opera sob o paradigma de árvore DOM (Document Object Model), onde cada elemento é um nó hierarquicamente relacionado.
- **Aplicação na Calculadora:** Definiria o container principal, o elemento de display (input ou div), e a matriz de botões (teclado virtual). Cada botão seria um nó com atributos identificadores (id, class, data-value).

**CSS (Cascading Style Sheets) - A Dermis Visual:**
- **Conceito Teórico:** Linguagem de estilo declarativa que aplica o princípio da cascata, especificidade e herança. Opera através do CSSOM (CSS Object Model), transformando a estrutura em uma interface gráfica renderizada pelo motor de layout do navegador (Blink, Gecko, WebKit).
- **Aplicação na Calculadora:** Implementaria o grid de disposição dos botões (normalmente via CSS Grid ou Flexbox), tipografia do display, estados interativos (:hover, :active) e responsividade (media queries).

**JavaScript (ECMAScript) - O Sistema Nervoso:**
- **Conceito Teórico:** Linguagem de programação interpretada, single-threaded com event loop, baseada em protótipos (embora sintaxe moderna suporte classes). Opera sob o paradigma de programação dirigida por eventos (Event-Driven Programming).
- **Aplicação na Calculadora:** Gerenciaria o estado da aplicação (operando atual, operando anterior, operador selecionado), escutaria eventos de interface (Event Listeners), e executaria a lógica de cálculo aritmético e manipulação do DOM.

## 2.2. Padrões Arquiteturais em Micro-Aplicações

**Arquitetura em Camadas (Layered Architecture):**
Em uma calculadora, tipicamente observamos três camadas distintas:
1. **Camada de Apresentação:** Responsável pela renderização visual e captura de inputs (DOM + CSS).
2. **Camada de Lógica de Negócio:** Processamento das operações matemáticas e validações.
3. **Camada de Dados/Estado:** Armazenamento temporário dos valores durante a operação (gerenciamento de estado).

**Padrão MVC (Model-View-Controller) e Variações:**
- **Model:** Representa os dados (valor atual, valor anterior, histórico de operações).
- **View:** Representação visual (o display e os botões renderizados).
- **Controller:** Intermediário que processa inputs do usuário, atualiza o Model e reflete na View.

**Fluxo de Eventos e Loop de Eventos (Event Loop):**
O JavaScript no browser opera em um modelo não-bloqueante. Quando um usuário clica em um botão:
1. O evento é capturado pelo browser e colocado na callback queue.
2. O Event Loop verifica se a call stack está vazia.
3. O handler do evento é executado, potencialmente modificando o estado e o DOM.
4. O browser realiza o repaint/reflow se necessário.

## 4.2. Descrição Detalhada dos Componentes Esperados

### 4.2.1. Camada de Apresentação (Front-End Visual)

**Para não-técnicos:** Imagine a calculadora como um carro. Esta camada é a carroceria, o painel e os botões que você vê e toca. É puramente visual.

**Para técnicos:**
- **index.html** serviria como o ponto de entrada single-page application (SPA).
- Estrutura esperada:

  - Container principal com `role="application"` para acessibilidade (ARIA).
  - Elemento de display provavelmente seria um `<input type="text" readonly>` ou `<div>` com atributos de live region para screen readers.
  - Grid de botões organizados semanticamente, possivelmente dentro de um `<div role="group" aria-label="Teclado numérico">`.

### 4.2.2. Camada de Lógica e Controle

**Para não-técnicos:** Esta é a parte "inteligente". Quando você aperta "2 + 2", esta camada entende que deve guardar o primeiro "2", lembrar que a operação é adição, esperar o segundo "2", e então mandar calcular.

**Para técnicos:**
- **script.js** conteria:
  - **Gerenciamento de Estado:** Variáveis para manter o contexto da operação atual.
  - **Event Listeners:** Captura de `click` events nos botões e possivelmente `keydown` para suporte a teclado físico.

### 4.2.3. Camada de Estilização e UX

**Para não-técnicos:** Esta camada decide se a calculadora parece moderna ou antiga, se os botões são grandes ou pequenos, e se funciona bem no celular.

**Para técnicos:**
- **style.css** implementaria:
  - **Layout:** CSS Grid com `grid-template-columns: repeat(4, 1fr)` para os botões.
  - **Design System:** Variáveis CSS (`:root`) para cores primárias, secundárias, estados de erro.
  - **Interatividade:** Transições suaves em `transform` e `background-color` para feedback tátil visual.
  - **Responsividade:** Media queries para ajustar tamanhos em viewports menores que 480px.

# 7. ANÁLISE DE RISCOS E LIMITAÇÕES ATUAIS

## 7.1. Impossibilidade de Análise de Qualidade de Código

Sem acesso aos artefatos, não é possível avaliar:
- **Cyclomatic Complexity:** Quantidade de caminhos independentes através do código.
- **Cognitive Complexity:** Dificuldade de compreensão humana do código.
- **Code Coverage:** Percentual de lógica testável vs implementada.
- **Technical Debt:** Identificação de código duplicado, magic numbers, ou falta de modularização.

## 7.2. Impossibilidade de Análise de Segurança

Não é possível verificar:
- **XSS (Cross-Site Scripting):** Se o display utiliza `innerHTML` (vulnerável) vs `textContent` (seguro) para exibir resultados.
- **Prototype Pollution:** Se o código modifica objetos nativos do JavaScript.
- **Input Validation:** Se há sanitização adequada de entradas para prevenir injection.

## 7.3. Impossibilidade de Análise de Performance

Não é possível determinar:
- **Time Complexity:** Eficiência dos algoritmos de cálculo (embora para uma calculadora básica seja tipicamente O(1)).
- **Memory Usage:** Uso de closures que podem causar memory leaks.
- **Reflow/Repaint:** Eficiência das atualizações de DOM.
