## Visão Geral do Projeto

O projeto **Dados** é uma solução robusta de ingestão, transformação e persistência de dados judiciais, extraídos de fontes NoSQL (MongoDB) e carregados em um data lake moderno baseado em Apache Iceberg. Utilizando Apache Spark como motor de processamento distribuído, o projeto garante que dados complexos e aninhados sejam convertidos em estruturas tabulares, otimizadas para análise, governança e escalabilidade.

O pipeline é projetado para ser incremental, eficiente e altamente configurável, permitindo fácil adaptação a mudanças de schema e requisitos de negócio. Cada etapa do processo é explicitamente definida, garantindo rastreabilidade, qualidade e integridade dos dados.

---

## Contexto Teórico da Arquitetura

### 1. **ETL (Extract, Transform, Load) Moderno**

- **Conceito:** Processo de extração de dados de múltiplas fontes, transformação para um modelo analítico (tipagem, flattening, normalização) e carregamento em um repositório central (Data Lake).
- **Aplicação no Projeto:** O código implementa um pipeline ETL onde:
    - **Extract:** Dados são lidos do MongoDB.
    - **Transform:** Estruturação, tipagem e limpeza via PySpark.
    - **Load:** Persistência em tabelas Iceberg (formato Parquet).

### 2. **Data Lake com Apache Iceberg**

- **Conceito:** Iceberg é um formato de tabela open source para data lakes, que traz recursos de versionamento, schema evolution, particionamento avançado e performance otimizada para grandes volumes de dados.
- **Aplicação no Projeto:** Cada coleção MongoDB é convertida em uma tabela Iceberg, garantindo governança, rastreabilidade e escalabilidade.

### 3. **Processamento Distribuído com Apache Spark**

- **Conceito:** Spark permite processamento paralelo de grandes conjuntos de dados, utilizando DataFrames para operações analíticas e transformações complexas.
- **Aplicação no Projeto:** Toda transformação, flattening e tipagem é realizada via PySpark, garantindo performance e escalabilidade.

### 4. **Schema Enforcement e Flattening**

- **Conceito:** Dados semi-estruturados (JSON, BSON) frequentemente possuem campos aninhados. Flattening é o processo de transformar essa estrutura em um modelo plano, compatível com bancos relacionais e data lakes.
- **Aplicação no Projeto:** Cada campo aninhado é explicitamente mapeado para uma coluna plana, com nomes padronizados.

### 5. **Incrementalidade e CDC (Change Data Capture)**

- **Conceito:** Processos incrementais carregam apenas dados novos ou alterados, otimizando recursos e evitando duplicidade.
- **Aplicação no Projeto:** Campos como `created_at` e `updated_at` são usados para identificar e processar apenas registros relevantes em cada execução.

## Arquitetura do Sistema

O pipeline de dados é modular, com cada arquivo Python representando um job ETL específico para uma coleção MongoDB. O fluxo geral segue:

- **Extração:** Leitura dos dados do MongoDB, utilizando schemas explícitos para garantir tipagem correta.
- **Transformação:** Flattening, renomeação e tipagem dos campos, preparando os dados para o modelo relacional.
- **Carregamento:** Escrita dos dados em tabelas particionadas no data lake (Iceberg), otimizadas para queries analíticas.

## Análise Detalhada dos Componentes

### Análise Exaustiva do Arquivo: `feitos.py`

#### 1. Importações

```python
from pyspark.sql import DataFrame
from pyspark.sql.functions import col
from pyspark.sql.types import StructType, StructField, StringType, BooleanType, LongType, IntegerType, TimestampType
```
- **Função:** Importa tipos e funções essenciais para manipulação de DataFrames Spark, definição de schemas e transformação de dados.

#### 2. Função de Transformação: `_transform`

**Assinatura:**
```python
def _transform(df: DataFrame) -> DataFrame:
```
- **Entrada:** DataFrame lido do MongoDB, estruturado conforme `mongodb_schema`.
- **Saída:** DataFrame plano, com colunas renomeadas e tipadas, pronto para persistência no Iceberg.

**Detalhamento do Processo:**
- **Flattening:** Campos aninhados como `classe.ancestral` tornam-se colunas planas `classe_ancestral`.
- **Renomeação:** Padroniza nomes para evitar inconsistências (`flgPrioridadePromotor` → `flgprioridadepromotor`).
- **Conversão de Tipos:** Exemplo: `versao` é convertido explicitamente para inteiro.
- **Seleção:** Apenas campos relevantes são mantidos.

**Exemplo de Transformação:**
```python
df.select(
    col("classe.ancestral").alias("classe_ancestral"),
    col("classe.superclasse").alias("classe_superclasse"),
    col("classe.nome").alias("classe_nome"),
    col("versao").cast("int").alias("versao"),
    ...
)
```
- **Didático:** Imagine um formulário digital com seções (aninhados) e campos. O flattening é como desmontar o formulário em uma planilha, onde cada coluna é um campo único.

#### 3. Dicionário de Configuração: `transformer_config`

Centraliza todos os parâmetros do job ETL.

| Chave                       | Descrição                                                                                  |
|-----------------------------|-------------------------------------------------------------------------------------------|
| `job_name`                  | Nome do job, usado para logging e auditoria                                               |
| `spark_master`              | Endereço do cluster Spark                                                                 |
| `mongodb_database_name`     | Nome do banco MongoDB de origem                                                           |
| `mongodb_collection_name`   | Nome da coleção MongoDB                                                                   |
| `iceberg_catalog_name`      | Catálogo Iceberg onde será criada a tabela                                                |
| `iceberg_namespace`         | Namespace do banco de dados Iceberg                                                       |
| `iceberg_table_name`        | Nome da tabela Iceberg                                                                    |
| `mongodb_schema`            | Definição detalhada do schema de origem (com campos aninhados)                            |
| `query_create_table`        | Comando SQL para criação da tabela de destino, refletindo o modelo plano                  |
| `filter_columns`            | Colunas usadas para filtragem incremental                                                 |
| `refresh_rate_history`      | Frequência de atualização histórica                                                       |
| `refresh_rate`              | Frequência de atualização incremental                                                     |
| `first_run_date`            | Data de referência para primeira execução                                                 |
| `transform`                 | Referência à função de transformação                                                      |

**Exemplo de Schema Detalhado:**
```python
StructField("classe", StructType([
    StructField("ancestral", StringType(), True),
    StructField("superclasse", StringType(), True),
    ...
]), True),
```
- **Nota:** Cada campo, mesmo aninhado, é explicitamente tipado.

**Exemplo de Criação de Tabela Iceberg:**
```sql
CREATE TABLE IF NOT EXISTS mpmg_raw.db_rw_mpe.tb_rw_feitos (
    classe_ancestral STRING,
    classe_superclasse STRING,
    ...
    job_dh_ref TIMESTAMP
)
USING iceberg
PARTITIONED BY (days(job_dh_ref))
LOCATION 's3://datalake/mpmg_raw/db_rw_mpe/tb_rw_feitos'
TBLPROPERTIES (
    'write.format.default' = 'parquet',
    'write.parquet.compression-codec' = 'zstd',
    ...
)
```

#### 4. Fluxo de Dados

1. **Leitura:** Spark lê dados da coleção MongoDB, aplicando o schema.
2. **Transformação:** `_transform` realiza flattening, renomeação e tipagem.
3. **Filtragem Incremental:** Apenas registros novos/alterados (via `created_at`/`updated_at`) são processados.
4. **Persistência:** DataFrame é salvo em tabela Iceberg particionada por data, em formato Parquet.

## Pontos de Atenção e Boas Práticas

- **Validação de Dados:** Implementar checagens pós-leitura e pós-transformação para garantir que todos os campos obrigatórios estejam presentes e corretos.
- **Monitoramento:** Utilizar o `job_name` e logs detalhados para rastrear execuções e identificar falhas rapidamente.
- **Evolução de Schema:** Mudanças no MongoDB exigem atualização do schema, da função de transformação e da tabela de destino.
- **Tratamento de Erros:** Exceções como “Expecting property name enclosed in double quotes” geralmente indicam problemas de serialização/deserialização JSON. O uso de schemas explícitos no Spark minimiza, mas não elimina totalmente, a possibilidade de dados malformados. Recomenda-se implementar validações e sanitização de dados de entrada.

## Apêndice: Referências de Erros e Correções

- **Exception “Expecting property name enclosed in double quotes”:**
    - Origem: Dados JSON/BSON malformados ao serem lidos do MongoDB.
    - Correção: Garantir que o schema Spark seja seguido à risca e implementar validação dos dados antes e após a leitura.
