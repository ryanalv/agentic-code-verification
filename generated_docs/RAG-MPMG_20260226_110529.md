# Documentação Técnica Oficial – Projeto RAG-MPMG

---

## 1. Visão Geral

O **RAG-MPMG** é um sistema de automação jurídica para o Ministério Público de Minas Gerais, projetado para processar, gerar e modificar minutas de documentos, realizar OCR (Reconhecimento Óptico de Caracteres) e armazenar informações jurídicas de maneira robusta e escalável. O sistema integra múltiplos bancos de dados, motores de IA, armazenamento vetorial e gerencia tarefas assíncronas e recorrentes. Toda a operação é exposta via API RESTful, permitindo fácil integração com outros sistemas e usuários.

### Para Leitores Não Técnicos

Imagine um "escritório digital" que processa milhares de documentos jurídicos automaticamente, usando inteligência artificial para gerar textos, analisar arquivos e organizar informações. Ele funciona como um grande centro de triagem, onde cada documento passa por etapas de validação, processamento e armazenamento, tudo supervisionado por mecanismos de controle de tarefas e logs para garantir rastreabilidade e segurança.

## 2. Contexto Teórico da Arquitetura

### 2.1. Recuperação Aumentada por Geração (RAG)

**RAG** é um padrão arquitetural que combina busca em bancos de dados/documentos com geração de texto por IA. O sistema primeiro recupera informações relevantes (ex: minutas, modelos, dados judiciais) e depois usa um modelo de linguagem para gerar ou modificar documentos com base nesse contexto.

#### Aplicação no RAG-MPMG

- O sistema busca documentos/modelos em MongoDB, SQL Server e HDFS.
- Com essas informações, monta prompts para IA (OpenAI, Azure, Gemini).
- A IA gera ou modifica minutas, que são então armazenadas e disponibilizadas ao usuário.

### 2.2. Bancos Vetoriais

**Bancos vetoriais** (ex: Qdrant, OpenAI Vector Store) armazenam representações matemáticas (embeddings) de textos/documentos, permitindo buscas semânticas rápidas e precisas.

#### Aplicação no RAG-MPMG

- Integração com Qdrant/OpenAI para upload, listagem e associação de arquivos.
- Permite encontrar documentos similares, associar minutas e facilitar pesquisas jurídicas.

### 2.3. Gerenciamento de Filas e Tarefas Assíncronas

O sistema usa **Redis** como broker de filas para distribuir tarefas de processamento (ex: geração de minutas, OCR) entre múltiplos workers. Isso garante escalabilidade e evita sobrecarga.

#### Aplicação

- Cada requisição é transformada em uma tarefa enfileirada.
- Workers consomem as filas, processam as tarefas e atualizam o status.
- Fallback entre provedores de IA é gerenciado por locks em Redis.

### 2.4. Scheduler de Tarefas Recorrentes

**Scheduler** permite agendar tarefas periódicas, como limpeza de arquivos expirados e chats, garantindo manutenção automática do sistema.

### 2.5. Padrão Hexagonal/Layered

O sistema separa responsabilidades em camadas:

- **API Layer**: Interface com usuários/sistemas externos.
- **Core Layer**: Infraestrutura, utilitários, clientes externos.
- **Repositories**: Acesso a dados.
- **Services**: Regras de negócio e processamento.
- **Schemas**: Contratos de dados.

## 3. Tecnologias Utilizadas

- **FastAPI**: Framework para APIs RESTful, rápido e com tipagem forte.
- **Ray**: Suporte a execução distribuída (preparado para uso futuro).
- **Redis**: Gerenciamento de filas e status de jobs.
- **MongoDB**: Persistência de documentos e minutas.
- **SQL Server**: Armazenamento de dados judiciais.
- **HDFS**: Armazenamento de arquivos binários.
- **Qdrant/OpenAI Vector Store**: Busca semântica de documentos.
- **Neo4j**: Banco de grafos para relacionamentos complexos.
- **OpenAI/Azure/Google Gemini**: Provedores de IA.
- **Pydantic**: Validação de contratos de dados.
- **RotatingFileHandler**: Logging rotativo.
- **schedule**: Agendamento de tarefas.
- **Docker Compose**: Orquestração de containers.

## 5. Análise Detalhada de Componentes

### 5.1. main.py

- **Carrega variáveis de ambiente** (.env).
- Instancia configuração global (`Config`).
- Cria aplicação FastAPI com metadados.
- Inclui rotas da API v1.

#### Fluxo de Execução

1. Carregar .env
2. Instanciar Config
3. Criar FastAPI
4. Incluir rotas v1

#### Casos de Erro

- Falha ao carregar .env pode causar erros silenciosos de configuração.

---

### 5.2. docker-compose.yml

- Orquestra Qdrant (banco vetorial) e Neo4j (grafo).
- Define volumes, healthchecks e plugins.

#### Limitações

- Não há balanceamento de carga ou auto scaling.

---

### 5.3. app/core/config.py

- **Classe Config**: Centraliza variáveis de ambiente.
- Define parâmetros de bancos, IA, Redis, logging, thresholds.
- Métodos para montar ambiente Ray.

#### Decisão

- Toda configuração via .env, facilitando portabilidade.

#### Casos de Erro

- Falta de variáveis pode causar falhas silenciosas.

---

### 5.4. app/core/logger.py

- **Classe Logger**: Logging rotativo, níveis configuráveis, formato detalhado.
- Métodos: debug, info, warning, error, critical, exception.

#### Tradeoff

- Pode perder logs em crashes severos.

---

### 5.5. app/core/queue.py

- **Classe QueueManager**: Gerencia filas Redis, status de jobs, estatísticas.
- Suporte a múltiplas prioridades.
- Métodos: enqueue_task, dequeue_task, get_job_status, queue_stats, clear_queue, requeue_processing_jobs.

#### Pseudocódigo enqueue_task

```python
if job_id not exists:
    job_id = uuid4()
set_job_status(job_id, 'pending')
serialize task
rpush to Redis queue
return job_id
```

#### Casos de Erro

- Falha de conexão Redis, corrupção de dados, perda de jobs.

---

### 5.6. app/api/v1.py

- Define endpoints RESTful.
- Valida parâmetros, enfileira tarefas, retorna StandardResponse.

#### Endpoints

- /v1/ – Healthcheck
- /v1/gerar-minuta/{id}
- /v1/modificar-minuta/{id}
- /v1/ocr-documentos-processo
- /v1/queue/status/{job_id}
- /v1/queue/info

#### Casos de Erro

- Falha ao enfileirar tarefas, parâmetros inválidos, indisponibilidade Redis.

---

### 5.7. app/schemas/geral.py

- **Classe StandardResponse**: Contrato de resposta padrão.
- Métodos: create, to_json_response, response.

#### Tradeoff

- Padronização, mas pode mascarar erros específicos.

---

### 5.8. app/core/client/*

- Clientes para Azure OpenAI, Google Gemini, OpenAI.
- Encapsulam autenticação, configuração, geração de texto, upload/listagem/deleção de arquivos.

#### Decisão

- Facilita fallback e troca de provedores.

---

### 5.9. app/core/db/*

- Abstração de acesso a MongoDB, SQL Server, HDFS, Redis.
- CRUD, busca, upsert, execução de queries.

#### Tradeoff

- Não há controle transacional entre bancos/documentos.

---

### 5.10. app/repositories/*

- Encapsulam regras de negócio de acesso a dados.
- Combinam múltiplas fontes, DTOs e schemas.

---

### 5.11. app/services/gpt_service.py

- Orquestra geração de documentos via IA.
- Monta prompts, fallback entre OpenAI e Gemini, atualiza status e persistência.

#### Fluxo

1. Monta prompt.
2. Tenta OpenAI.
3. Se falha, ativa Gemini e lock em Redis.
4. Atualiza documento no MongoDB.

#### Pseudocódigo

```python
montar_prompt()
try:
    if not lock_gemini:
        gerar_OpenAI()
    else:
        gerar_Gemini()
except Exception:
    ativar_lock_gemini()
    gerar_Gemini()
atualizar_documento()
return id_documento
```

#### Casos de Erro

- Falha de provedores, inconsistência de respostas, lock não liberado.

---

### 5.12. app/services/ocr_service.py

- Processamento de OCR para PDFs, HTML, DOCX.
- Usa Surya OCR, BeautifulSoup, python-docx.

---

### 5.13. app/services/vector_store_service.py

- Upload de textos para vector store OpenAI.
- Remoção de arquivos expirados.
- Integração com MongoDB para controle de expiração.

---

### 5.14. app/core/scheduler.py

- Agendamento de tarefas recorrentes (limpeza, expiração).
- Logging dedicado.

## 6. Fluxos de Dados

### 6.1. Geração de Minuta

1. Usuário chama `/v1/gerar-minuta/{id}`.
2. API valida e enfileira tarefa em Redis.
3. Worker consome fila, busca documentos/modelos.
4. Monta prompt, chama IA.
5. Atualiza documento no MongoDB.
6. Usuário consulta status via `/v1/queue/status/{job_id}`.

### 6.2. OCR de Documentos

1. Usuário chama `/v1/ocr-documentos-processo`.
2. API enfileira tarefa.
3. Worker busca arquivos.
4. Executa OCR, atualiza status.

### 6.3. Fallback IA

1. Tenta OpenAI.
2. Se erro, ativa Gemini, registra lock em Redis.
3. Atualiza documento.

## 10. Riscos, Limitações e Tradeoffs

### Riscos

- Dependência de múltiplos serviços externos.
- Falhas silenciosas por variáveis de ambiente ausentes.
- Fallback IA pode mascarar erros ou inconsistências.
- Redis como ponto único de falha.
- Rotação de logs mal configurada pode perder informações críticas.
- Falta de controle transacional entre bancos.
- Manipulação de arquivos binários pode causar estouro de memória.

### Limitações

- Dependência de serviços externos corretamente configurados.
- Gerenciamento de filas não garante exactly-once.
- Provedores IA configurados via .env, sem hot reload.
- Fallback IA não compara respostas semanticamente.
- Expiração de jobs limitada ao TTL do Redis.
- API sem autenticação/autorização nativa.
- Orquestração docker-compose sem balanceamento de carga.

### Tradeoffs

- FastAPI: Agilidade e tipagem forte, sem suporte nativo a workers.
- Redis: Leve e rápido, não robusto como RabbitMQ/Celery.
- Status de jobs em Redis: Rápido, mas volátil.
- Fallback IA: Disponibilidade, mas respostas divergentes.
- Logger em disco: Simples, mas pode perder logs.
- Configuração via .env: Flexível, mas fonte de erro humano.

---

## 11. Casos de Erro

- Falha de conexão com Redis/MongoDB/SQL Server/HDFS gera logs e respostas de erro.
- Falha ao enfileirar/desenfileirar retorna erro explícito.
- Falha de provedores IA ativa fallback e lock.
- Falta de variáveis de ambiente pode causar falhas silenciosas.
- Corrupção de dados serializados pode impedir processamento.
