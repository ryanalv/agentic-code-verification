# Documentação Técnica do Projeto RAG-MPMG

## Visão Geral

O projeto RAG-MPMG é uma aplicação sofisticada projetada para implementar um pipeline de RAG (Retrieve and Generate) especificamente voltado ao processamento de documentos. O foco principal é lidar com normas internas do Ministério Público de Minas Gerais (MPMG). Baseada em uma arquitetura de microserviços, a aplicação usa FastAPI para construir uma interface REST eficaz e Ray para processamento paralelo e distribuído. As funcionalidades principais incluem OCR (Reconhecimento Óptico de Caracteres) de documentos e a geração automatizada de respostas a partir das informações extraídas.

## Tecnologias Utilizadas

- **FastAPI**: Um framework moderno para a construção de APIs com Python, reconhecido por sua alta performance e facilidade de uso para criar endpoints intuitivos.
  
- **Ray**: Biblioteca poderosa para processamento paralelo e distribuído, que aumenta a capacidade de gerenciar tarefas e computação em grande escala eficientemente.
  
- **MongoDB**: Banco de dados NoSQL usado para armazenamento e recuperação robusta de documentos.
  
- **Redis**: Sistema de cache e estrutura de dados em memória, usado para gerenciar filas de processamento e estados das tarefas no RAG-MPMG.
  
- **Pydantic**: Utilizado para validação e criação de modelos de dados, garantindo integridade e segurança dos dados processados.
  
- **Surya OCR**: Tecnologia empregada para extração de texto a partir de documentos não estruturados, crucial na transformação de documentos em fontes de dados úteis.
  
- **PyMuPDF**: Biblioteca para manipulação e leitura de arquivos PDF, compatível com o Surya OCR para melhorar o reconhecimento de texto.
  
- **HDFS**: Hadoop Distributed File System, responsável por armazenar grandes volumes de dados com segurança e eficiência.

## Arquitetura

A arquitetura do RAG-MPMG é composta por múltiplas camadas, cada uma desempenhando funções vitais no funcionamento do sistema:

1. **Camada de Apresentação**: Implementada com rotas do FastAPI, esta camada disponibiliza operações que expõem os recursos da aplicação aos usuários finais.

2. **Camada de Negócio**: Executa a lógica principal da aplicação, orquestrando operações entre as rotas e repositórios de dados para cumprir as demandas do usuário.

3. **Camada de Dados**: Utiliza MongoDB para armazenamento resiliente de dados e Redis para gerenciamento eficiente de filas e status de tarefas.

4. **Camada de Serviços**: Interage diretamente com APIs externas e bibliotecas, fornecendo a funcionalidade necessária para integração e processamento externo.

### Estrutura do Código

O arquivo `v1.py` dentro do diretório `app/api/` representa o núcleo dos endpoints da aplicação. Aqui está a análise detalhada de cada seção do código.

from fastapi import APIRouter, Path, Body

As importações estabelecem as dependências essenciais, como módulos FastAPI para criação de rotas e tipos, além de utilitários para logging e gestão de filas.

#### Criação do Router

```python
router = APIRouter(
    prefix="/v1",
    tags=["v1"],
    responses={404: {"description": "Not found"}},

```

O `APIRouter` é configurado para agrupar logicamente todos os endpoints da versão "/v1" da API.

#### Logger

```python
logger = Logger("app.api.v1")
logger.info("Iniciando API v1.")
```

Um objeto `Logger` é instanciado para registrar informações e erros, proporcionando melhor rastreamento e auditabilidade durante a execução da API.

#### QueueManager

```python
queue_manager = QueueManager()
```

O `QueueManager` gerencia as filas de tarefas, permitindo enfileiramento eficiente e consulta do estado das tarefas.

### Endpoints da API

#### 1. Root Endpoint `GET /`

```python
@router.get("/")
def root() -> StandardResponse:
    return StandardResponse.response(message="Hello World")
```

- **Objetivo**: Endpoint básico usado para verificar o estado operacional da API, retornando uma mensagem padrão "Hello World".

#### 2. Endpoint `GET /gerar-minuta/{id}`

```python
@router.get("/gerar-minuta/{id}")
def gerar_minuta_endpoint(
    id: str = Path(..., description="ID único do modelo a ser gerado", example="685c6a9c890098de44ea7e0f")
) -> StandardResponse:
    try:
        job_id = queue_manager.enqueue_task({
            "task": "processar_pipeline_minuta_task",
            "args": (id,)
        }, get_prioridade_task_minuta(id))
        return StandardResponse.response(data={"job_id": job_id})
    except Exception as e:
        error_message = f"Erro ao criar task de geração de minuta: {e}"
        logger.error(error_message)
        return StandardResponse.response(message=error_message, status_code=500)
```

- **Objetivo**: Gere uma minuta baseada em um ID específico. Usa `QueueManager` para enfileirar a tarefa.

- **Tratamento de Erros**: Loga erros e retorna uma mensagem de erro em caso de exceções.

#### 3. Endpoint `GET /modificar-minuta/{id}`

```python
@router.get("/modificar-minuta/{id}")
def modificar_minuta_endpoint(
    id: str = Path(..., description="ID único do modelo a ser gerado", example="685c6a9c890098de44ea7e0f")
) -> StandardResponse:
    try:
        job_id = queue_manager.enqueue_task({
            "task": "processar_pipeline_minuta_task",
            "args": (id, True)
        }, get_prioridade_task_minuta(id))
        return StandardResponse.response(data={"job_id": job_id})
    except Exception as e:
        error_message = f"Erro ao criar task de modificação de minuta: {e}"
        logger.error(error_message)
        return StandardResponse.response(message=error_message, status_code=500)
```

- **Objetivo**: Similar ao endpoint de geração, mas modifica uma minuta existente, indicado pelo parâmetro `True`.

#### 4. Endpoint `POST /ocr-documentos-processo`

```python
@router.post("/ocr-documentos-processo")
def ocr_documentos_processo_endpoint(
    id: str = Body(..., description="ID do processo", example="23597"),
    tipo: Literal['sru', 'mpe', 'mpeEletronico'] = Body(..., description="Origem do processo", example="sru"),
    login: str = Body(..., description="Login do usuário", example="liberiosilva"),
    arquivos: Optional[list[str]] = Body(None, description="Arquivos a serem processados", example=["1234567890", "1234567891"]),
    vector_store_id: Optional[str] = Body(None, description="ID do vector store", example="vs_n79bnskaWzhAq1gHvkEuAqbz"),
) -> StandardResponse:
    try:
        job_id = queue_manager.enqueue_task({
            "task": "processar_ocr_documentos_processo_task",
            "args": (id, tipo, login, arquivos, vector_store_id)
        }, get_prioridade_taks_documentos(len(arquivos)))
        return StandardResponse.response(data={"job_id": job_id})
    except Exception as e:
        error_message = f"Erro ao criar task de OCR de documentos de processo: {e}"
        logger.error(error_message)
        return StandardResponse.response(message=error_message, status_code=500)
```

- **Objetivo**: Inicia OCR em documentos de um processo, configurando uma tarefa apropriada à fila.

#### 5. Endpoint `GET /queue/status/{job_id}`

```python
@router.get("/queue/status/{job_id}")
def get_job_status(
    job_id: str = Path(..., description="ID do job no Redis")
) -> StandardResponse:
    try:
        status = queue_manager.get_job_status(job_id)
        return StandardResponse.response(data=status)
    except Exception as e:
        error_message = f"Erro ao buscar status do job: {e}"
        logger.error(error_message)
        return StandardResponse.response(message=error_message, status_code=500)
```

- **Objetivo**: Permite consultas ao status de uma tarefa específica baseada em seu ID no Redis.

#### 6. Endpoint `GET /queue/info`

```python
@router.get("/queue/info")
def get_queue_info() -> StandardResponse:
    try:
        high = queue_manager.queue_stats(HIGH_PRIORITY_QUEUE)
        default = queue_manager.queue_stats(DEFAULT_QUEUE)
        low = queue_manager.queue_stats(LOW_PRIORITY_QUEUE)
        info = {**high, **default, **low}
        return StandardResponse.response(data=info)
    except Exception as e:
        error_message = f"Erro ao buscar informações das filas: {e}"
        logger.error(error_message)
        return StandardResponse.response(message=error_message, status_code=500)
```

- **Objetivo**: Retorna estatísticas das filas incluindo o número de jobs em cada uma.

## Fluxos de Dados

### Fluxo de Geração de Minutas
1. **Requisição do Usuário**: O usuário acessa o endpoint `/gerar-minuta/{id}`.
2. **Enfileiramento**: O `QueueManager` enfileira a tarefa.
3. **Processamento**: Um worker processa e gera a minuta.

### Fluxo de OCR
1. **Inicialização**: Via endpoint `/ocr-documentos-processo`.
2. **Preenchimento da Fila**: Tarefas de OCR são enfileiradas para processamento paralelo.
3. **Status em Tempo Real**: Consultas podem ser feitas para verificar o progresso.

### Fluxo de Consultas de Status
1. **Verificação de andamento**: Os usuários podem consultar `/queue/status/{job_id}` para status atualizado das tarefas.
