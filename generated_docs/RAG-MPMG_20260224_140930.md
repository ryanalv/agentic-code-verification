# Documentação Técnica do Projeto RAG-MPMG

## Visão Geral

O projeto RAG-MPMG é uma aplicação destinada ao processamento de documentos legais, especificamente normas internas do Ministério Público de Minas Gerais (MPMG). Ele implementa um pipeline de RAG (Retrieval-Augmented Generation) para facilitar a extração e análise de informações. A aplicação adota uma arquitetura de microserviços, com uma API REST para interações, permitindo operações como OCR (Reconhecimento Óptico de Caracteres), extração de metadados e busca vetorial.

## Tecnologias

O projeto utiliza diversas tecnologias e bibliotecas modernas:

- **FastAPI**: Framework para construção de APIs RESTful de alto desempenho.
- **Ray**: Framework para computação distribuída, utilizado para processamento paralelo de tarefas.
- **Surya OCR**: Biblioteca para reconhecimento óptico de caracteres.
- **FAISS**: Biblioteca para busca vetorial e indexação de dados.
- **MongoDB**: Banco de dados NoSQL para armazenamento de documentos e metadados.
- **SQL Server**: Banco de dados relacional para operações específicas.
- **HDFS**: Sistema de arquivos distribuído para armazenamento de grandes volumes de dados.

## Arquitetura

A arquitetura do projeto é baseada em microserviços, com comunicação entre serviços via chamadas HTTP utilizando a arquitetura REST. A estrutura do projeto é organizada em diretórios que segregam funcionalidades em módulos específicos:

## Contexto Teórico da Arquitetura

### RAG (Retrieval-Augmented Generation)

RAG é uma técnica que combina recuperação de informações com geração de texto. Ela utiliza um modelo de recuperação para buscar documentos relevantes e um modelo de geração para sintetizar respostas baseadas nesses documentos. No contexto do RAG-MPMG, essa técnica é aplicada para processar e analisar documentos legais, permitindo a extração de informações relevantes de grandes volumes de texto.

### Microserviços

A arquitetura de microserviços divide a aplicação em serviços pequenos e independentes, cada um responsável por uma funcionalidade específica. Isso facilita a escalabilidade e a manutenção, permitindo que cada serviço seja desenvolvido, implantado e escalado de forma independente.

### Computação Distribuída com Ray

Ray é um framework para computação distribuída que permite o processamento paralelo de tarefas. No RAG-MPMG, ele é utilizado para distribuir a carga de trabalho entre múltiplos nós, melhorando o desempenho e a eficiência do processamento de documentos.

### `app/core/queue.py`

Este arquivo implementa a classe `QueueManager`, que gerencia as filas de tarefas.

```python
class QueueManager:
    """Gerenciador de fila Redis para processamento Ray com suporte a múltiplas filas e prioridades"""
    
    def __init__(self):
        self.config = Config()
        self.redis_client = Redis(
            host=self.config.queue_redis_host,
            port=self.config.queue_redis_port,
            db=self.config.queue_redis_db
        )
    
    def enqueue_task(self, data: Dict[str, Any], queue: str = DEFAULT_QUEUE) -> str:
        """
        Adiciona uma tarefa à fila especificada com ID único
        
        Args:
            data: Dicionário com os dados da tarefa
            queue: Nome da fila (padrão: 'default')
            
        Returns:
            ID da tarefa se adicionada com sucesso, None caso contrário
        """
        try:
            if 'job_id' not in data:
                data['job_id'] = str(uuid.uuid4())
            
            job_id = data['job_id']
            self._set_job_status(job_id, STATUS_PENDING, queue, data)

            queue_name = self._get_queue_name(queue)
            self.redis_client.rpush(queue_name, json.dumps(data))
            
            return job_id
        except (RedisError, json.JSONDecodeError) as e:
            print(f"Erro ao enfileirar tarefa: {e}")
            return None
```

**Análise:**

- **Gerenciamento de Filas**: A classe `QueueManager` é responsável por gerenciar as filas de tarefas utilizando Redis, permitindo que a aplicação suporte processamento assíncrono.
- **Enfileiramento de Tarefas**: O método `enqueue_task` adiciona uma nova tarefa à fila, atribuindo um ID único e salvando o status da tarefa. Isso é crucial para o controle de fluxo da aplicação.

### `app/schemas/geral.py`

Este arquivo define o modelo de resposta padrão para as rotas da API.

```python
class StandardResponse(BaseModel):
    """
    Resposta padrão para todas as rotas.
    """
    status: str = Field(default="success", description="Status da resposta")
    success: bool = Field(default=True, description="Sucesso da resposta")
    message: str = Field(default="Solicitação processada com sucesso", description="Mensagem da resposta")
    data: dict = Field(default={}, description="Dados da resposta")
    error: str = Field(default="", description="Erro da resposta")

    @classmethod
    def response(cls, data: Optional[dict] = None, message: Optional[str] = None, error: Optional[str] = None, status_code: int = 200) -> JSONResponse:
        """
        Cria e retorna diretamente um JSONResponse.
        """
        response = cls.create(data=data, message=message, error=error, status_code=status_code)
        return response.to_json_response()
```

**Análise:**

- **Modelagem de Respostas**: A classe `StandardResponse` define uma estrutura padrão para as respostas da API, facilitando a consistência nas respostas e a manipulação de erros.
- **Método de Resposta**: O método `response` permite criar e retornar uma resposta JSON formatada de maneira consistente, melhorando a legibilidade e a manutenção do código.

```python
def ocr_documento_pdf(binario: bytes) -> str:    
    recognition_predictor = RecognitionPredictor()
    detection_predictor = DetectionPredictor()
    
    doc = fitz.open(stream=binario, filetype='pdf')
    imagens = []
    for page in doc:        
        pix = page.get_pixmap(dpi=300)
        img = Image.frombytes("RGB", [pix.width, pix.height], pix.samples)
        imagens.append(img)
    
    predictions = recognition_predictor(imagens, det_predictor=detection_predictor)
    texto = ' '.join(
        text_line.text for ocr_result in predictions for text_line in ocr_result.text_lines
    )
    return texto
```

- **Flexibilidade**: A implementação permite a adição de novos formatos de documentos facilmente, pois cada formato pode ter sua própria função de OCR.

## Fluxos de Dados

O fluxo de dados na aplicação RAG-MPMG é organizado em etapas bem definidas:

1. **Recepção de Solicitação**: O usuário faz uma solicitação através da API, que é recebida pelo FastAPI.
2. **Processamento de Tarefas**: Dependendo do endpoint chamado, a solicitação pode acionar tarefas de OCR, geração de modelos, ou consultas a documentos.
3. **Gerenciamento de Filas**: As tarefas são enfileiradas utilizando o `QueueManager`, que gerencia o status e a execução das tarefas.
4. **Execução de Tarefas**: As tarefas são processadas, podendo envolver chamadas a serviços externos (como OCR ou APIs de IA).
5. **Retorno de Resposta**: Após o processamento, uma resposta padrão é gerada e retornada ao usuário.
