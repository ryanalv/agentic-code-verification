# Documentação Técnica do Projeto RAG-MPMG

## Visão Geral

O projeto **RAG-MPMG** é concebido para processamento avançado de documentos legais, utilizando um paradigma de Arquitetura de Geração Aumentada por Recuperação (RAG) que viabiliza a transformação de documentos em conhecimento estruturado e consultável. Este projeto emprega reconhecimento óptico de caracteres (OCR) e modelos de linguagem de ponta para efetivar a extração e geração de respostas textuais, aumentando a eficácia do processamento legal.

A estrutura do projeto adota uma arquitetura modular e diferenciada, dividindo-se em diversas camadas funcionais: serviços, repositórios, casos de uso, e interfaces API. Essa organização garante uma clara separação de preocupações, assegurando a fácil manutenção e expansibilidade do sistema.

## Tecnologias Utilizadas

O projeto implementa de forma estratégica um conjunto de tecnologias avançadas:

- **FastAPI**: Serve como o backbone da API REST, promovendo alta performance e simplicidade na definição de rotas e endpoints.
- **Ray**: Facilita a computação distribuída, permitindo operações de processamento em paralelo, reforçando a escalabilidade.
- **MongoDB**: Atua como um banco de dados NoSQL para a persistência de documentos e metadados não estruturados.
- **SQL Server**: Providencia a gestão de dados estruturados através de um banco de dados relacional confiável.
- **HDFS (Hadoop Distributed File System)**: Responsável por armazenar arquivos volumosos de dados, caracterizando-se por sua robustez em ambientes distribuídos.
- **Surya OCR**: Realiza o reconhecimento de caracteres em documentos escaneados, fornecendo dados textuais brutos para processos subsequentes.
- **Pydantic**: Assegura a validação rigorosa dos dados de entrada e saída, estabelecendo modelos bem definidos.
- **OpenAI e Google Gemini**: Integram modelos de Inteligência Artificial para processamento de linguagem natural e geração de respostas inteligentes, potencializando a precisão e relevância do sistema.

## Arquitetura

A arquitetura do sistema é fundamentada no padrão MVC (Model-View-Controller), facilitando a separação de responsabilidades:

- **Model (Modelo)**: Contempla a lógica de negócios, sustentada por classes localizadas nas camadas de serviços e repositórios.
- **View (Visão)**: Abrange estruturas que permitem a interação com os usuários, aqui representadas pelos endpoints API desenvolvidos via FastAPI.
- **Controller (Controle)**: Intermediário que gerencia a comunicação entre modelos e visões, implementando a lógica de controle.

A organização do diretório é estruturada conforme:

```
RAG-MPMG/
|-- app/
|   |-- api/           # Implementação de rotas
|   |-- core/          # Definição de configurações e utilitários do sistema
|   |-- repositories/  # Acesso a dados e integração com bancos
|   |-- schemas/       # Modelos de dados e validações
|   |-- services/      # Regras de negócios e processamento lógico
|   |-- usecases/      # Casos de uso específicos implementados
|   |-- workers/       # Gerenciamento de tarefas paralelas via Ray
|-- main.py            # Entrypoint da aplicação
```

## Contexto Teórico da Arquitetura

### Arquitetura RAG (Retrieval-Augmented Generation)

A arquitetura RAG combina a recuperação de documentos relevantes com a geração de conteúdo automatizado. Este sistema decorre de duas partes principais: Recuperação, onde documentos são eficientemente buscados a partir de um repositório; e Geração Aumentada, onde modelos de linguagem são aplicados para gerar respostas ou sumarizações baseadas nos documentos selecionados. Este método é particularmente útil em cenários onde o contexto é crítico, como o processamento de documentos legais, permitindo respostas contextualizadas e fundamentadas.

### Implementação no Projeto

No contexto do RAG-MPMG, a recuperação de documentos ocorre principalmente por meio de consultas ao MongoDB, onde documentos legais são armazenados e indexados. Em seguida, uma camada de IA, implementada com modelos da OpenAI e Google Gemini, trabalha para gerar respostas baseadas nos dados recuperados, garantindo um conteúdo preciso e relevante.

### Análise de Funções Especializadas

#### Arquivo `app/core/queue.py`

Essa seção esquadrinha funções centrais para o gerenciamento das filas dentro do contexto de processamento assíncrono com Redis.

##### Função: `enqueue_task`

```python
def enqueue_task(self, data: Dict[str, Any], queue: str = DEFAULT_QUEUE) -> str:
    try:
        # Gera um ID único se não foi fornecido
        if 'job_id' not in data:
            data['job_id'] = str(uuid.uuid4())

        job_id = data['job_id']
        
        # Salva status como "pending"
        self._set_job_status(job_id, STATUS_PENDING, queue, data)

        queue_name = self._get_queue_name(queue)
        self.redis_client.rpush(queue_name, json.dumps(data))
        
        return job_id
    except (RedisError, json.JSONDecodeError) as e:
        print(f"Erro ao enfileirar tarefa: {e}")
        return None
```

Esse bloco incorporado adorna detalhes críticos.

- **Geração de um `job_id` Único**: Explora `uuid.uuid4()` para garantir unicidade de ID das tarefas dentro da fila.
- **Armazenamento e Enfileiramento**: Utiliza `rpush` do Redis para enfileirar os dados, garantindo que a ordem de prioridade de entrada na fila respeite o contexto assíncrono do processamento distribuído.
- **Resiliência e Tratamento de Erros**: O uso de exceções (`RedisError` e `json.JSONDecodeError`) proporciona ao sistema robustez e capacidade de adaptação a falhas.

## Fluxos de Dados

### Fluxo de Operação

1. **Solicitação de API**: Inicia-se com a recepção de requisições do usuário via endpoints, direcionadas para funções específicas via roteadores.
2. **Interação com Fila**: Tarefas são organizadas no gerenciador de filas, integrando a lógica de prioridade para determinar a ordem de execução.
3. **Execução Parallelizada**: Com auxílio da biblioteca Ray, workers são distribuídos para processar as tarefas, assegurando eficiência e desempenho.
4. **Resposta Final**: Ao término do processamento, os resultados são assimilados na forma de `StandardResponse`, sendo enviados de volta ao cliente.

Essas interações sistemáticas promovem um ecossistema coordenado dentro do projeto, assegurando correta implantação e execução lógica do processamento legal.
