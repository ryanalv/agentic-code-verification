# Documentação Técnica do Projeto RAG-MPMG

## Visão Geral

O projeto RAG-MPMG é uma implementação sofisticada de um pipeline de Recuperação de Informação e Geração de Respostas (RAG) voltado para o processamento de normas internas do Ministério Público de Minas Gerais (MPMG). Este sistema é projetado para lidar com documentos, realizar OCR (Reconhecimento Óptico de Caracteres) e fornecer uma API RESTful para interação com esses serviços. A arquitetura modular do projeto facilita a manutenção e a escalabilidade, permitindo que novas funcionalidades sejam integradas sem comprometer a estabilidade do sistema.

## Tecnologias Utilizadas

O projeto RAG-MPMG faz uso de uma série de tecnologias e bibliotecas modernas para garantir eficiência, robustez e escalabilidade:

- **FastAPI**: Um framework moderno e de alto desempenho para a construção de APIs RESTful, que facilita a criação de serviços web rápidos e eficientes.
- **Ray**: Um framework para computação distribuída, que permite o processamento paralelo de tarefas, essencial para lidar com grandes volumes de dados.
- **Surya OCR**: Uma biblioteca especializada em reconhecimento óptico de caracteres, utilizada para extrair texto de imagens e documentos.
- **FAISS**: Uma biblioteca para busca vetorial e indexação, utilizada para operações de busca eficiente em grandes conjuntos de dados.
- **MongoDB**: Um banco de dados NoSQL utilizado para armazenamento de documentos e metadados, oferecendo flexibilidade e escalabilidade.
- **SQL Server**: Um banco de dados relacional utilizado para operações específicas que requerem transações ACID.
- **HDFS**: Um sistema de arquivos distribuído utilizado para armazenamento de grandes volumes de dados, garantindo alta disponibilidade e tolerância a falhas.

### Contexto Teórico da Arquitetura

#### Recuperação de Informação e Geração de Respostas (RAG)

O conceito de Recuperação de Informação e Geração de Respostas (RAG) combina técnicas de busca e processamento de linguagem natural para extrair informações relevantes de grandes volumes de dados e gerar respostas contextualizadas. No projeto RAG-MPMG, este conceito é aplicado para processar normas internas do MPMG, permitindo a recuperação eficiente de informações e a geração de respostas automatizadas.

#### Computação Distribuída com Ray

Ray é um framework que facilita a implementação de aplicações de computação distribuída. Ele permite que tarefas sejam executadas em paralelo, distribuindo a carga de trabalho entre múltiplos nós. No RAG-MPMG, Ray é utilizado para processar tarefas de OCR em paralelo, aumentando a eficiência e reduzindo o tempo de processamento.

#### Armazenamento e Busca com MongoDB e FAISS

MongoDB é um banco de dados NoSQL que oferece flexibilidade no armazenamento de documentos e metadados. FAISS, por outro lado, é uma biblioteca especializada em busca vetorial, permitindo a indexação e recuperação eficiente de dados em grandes volumes. No projeto, essas tecnologias são combinadas para armazenar os resultados do OCR e permitir buscas rápidas e eficientes.

### Arquivo `app/core/queue.py`

Este módulo implementa a lógica de gerenciamento de filas utilizando Redis.

```python
from redis import Redis
from redis.exceptions import RedisError
import json
import uuid
import time
from typing import Optional, Dict, Any, List, Tuple
from app.core.config import Config

class QueueManager:
    def __init__(self):
        self.config = Config()
        self.redis_client = Redis(
            host=self.config.queue_redis_host,
            port=self.config.queue_redis_port,
            db=self.config.queue_redis_db
        )

    def enqueue_task(self, data: Dict[str, Any], queue: str = DEFAULT_QUEUE) -> str:
        try:
            if 'job_id' not in data:
                data['job_id'] = str(uuid.uuid4())
            job_id = data['job_id']
            self._set_job_status(job_id, STATUS_PENDING, queue, data)
            queue_name = self._get_queue_name(queue)
            self.redis_client.rpush(queue_name, json.dumps(data))
            return job_id
        except (RedisError, json.JSONDecodeError) as e:
            print(f"Erro ao enfileirar tarefa: {e}")
            return None
```

#### Análise

1. **Gerenciamento de Filas**: A classe `QueueManager` é responsável por gerenciar tarefas em filas utilizando Redis, permitindo que a aplicação processe tarefas em background.

2. **Enfileiramento de Tarefas**: O método `enqueue_task` adiciona uma nova tarefa à fila, gerando um ID único para cada tarefa. Isso é fundamental para rastrear o status das tarefas.

3. **Tratamento de Erros**: O tratamento de exceções garante que erros durante a enfileiração não quebrem a aplicação, permitindo uma operação mais robusta.

### Arquivo `app/schemas/geral.py`

Este módulo define o modelo de resposta padrão para as rotas da API.

```python
from pydantic import BaseModel, Field
from typing import Optional
from fastapi.responses import JSONResponse

class StandardResponse(BaseModel):
    status: str = Field(default="success", description="Status da resposta")
    success: bool = Field(default=True, description="Sucesso da resposta")
    message: str = Field(default="Solicitação processada com sucesso", description="Mensagem da resposta")
    data: dict = Field(default={}, description="Dados da resposta")
    error: str = Field(default="", description="Erro da resposta")

    @classmethod
    def response(cls, data: Optional[dict] = None, message: Optional[str] = None, error: Optional[str] = None, status_code: int = 200) -> JSONResponse:
        response = cls.create(data=data, message=message, error=error, status_code=status_code)
        return response.to_json_response()
```

#### Análise

1. **Modelo de Resposta**: A classe `StandardResponse` define a estrutura padrão para as respostas da API, facilitando a consistência nas respostas.

2. **Método de Criação de Resposta**: O método `response` permite criar uma resposta JSON com base nos parâmetros fornecidos, simplificando a manipulação de respostas em diferentes rotas.

### Arquivo `app/services/ocr_service.py`

Este módulo implementa a lógica para realizar OCR em documentos.

```python
import fitz
from PIL import Image
from surya.recognition import RecognitionPredictor
from surya.detection import DetectionPredictor

def ocr_documento_pdf(binario: bytes) -> str:    
    recognition_predictor = RecognitionPredictor()
    detection_predictor = DetectionPredictor()
    
    doc = fitz.open(stream=binario, filetype='pdf')
    imagens = []
    for page in doc:        
        pix = page.get_pixmap(dpi=300)
        img = Image.frombytes("RGB", [pix.width, pix.height], pix.samples)
        imagens.append(img)
    
    predictions = recognition_predictor(imagens, det_predictor=detection_predictor)
    texto = ' '.join(
        text_line.text for ocr_result in predictions for text_line in ocr_result.text_lines
    )
    return texto
```

#### Análise

1. **Processamento de PDF**: A função `ocr_documento_pdf` utiliza a biblioteca `fitz` para abrir e processar arquivos PDF, extraindo imagens de cada página.

2. **Reconhecimento de Texto**: O uso de `RecognitionPredictor` e `DetectionPredictor` da biblioteca Surya permite realizar o OCR nas imagens extraídas, convertendo-as em texto.

3. **Retorno de Texto**: O texto extraído é retornado como uma string, pronto para ser utilizado em outras partes do sistema.

## Fluxos de Dados

O fluxo de dados no sistema RAG-MPMG pode ser descrito da seguinte forma:

1. **Recepção de Requisições**: A API recebe requisições HTTP que são roteadas para os manipuladores apropriados.
2. **Processamento de Tarefas**: As tarefas são enfileiradas utilizando o `QueueManager`, que as coloca em uma fila Redis.
3. **Execução de OCR**: As tarefas de OCR são processadas em background, utilizando Ray para permitir processamento paralelo.
4. **Armazenamento de Resultados**: Os resultados do OCR são armazenados em MongoDB, permitindo consultas e recuperação de dados.
5. **Resposta ao Usuário**: Após o processamento, as respostas são formatadas utilizando o modelo `StandardResponse` e enviadas de volta ao cliente.
