# Documentação Técnica do Projeto RAG-MPMG

## Visão Geral

O projeto **RAG-MPMG** (Retrieval-Augmented Generation for the Public Ministry of Minas Gerais) visa criar uma API sofisticada para a produção e manipulação de documentos utilizando técnicas avançadas de inteligência artificial (IA). A solução integra componentes de aprendizagem de máquina com uma arquitetura moderna baseada em microserviços, suportando a execução de tarefas assíncronas, armazenamento de dados em bases NoSQL e grafos, bem como integrações robustas com serviços de IA da Microsoft e Google.

## Tecnologias Utilizadas

### Linguagem de Programação

- **Python**: Optamos por Python devido à sua popularidade e extensa gama de bibliotecas voltadas para desenvolvimento em inteligência artificial e processamento de dados, além de suas capacidades de integração com serviços web via HTTP.

### Frameworks e Bibliotecas

- **FastAPI**: Este framework foi escolhido para a construção da API devido à sua velocidade e suporte a recursos nativos como validação de dados, documentação automática e tipagem estática.
- **Ray**: Utilizada para computação paralela e distribuída, Ray possibilita a execução de tarefas em larga escala de maneira eficiente.
- **Pydantic**: Utilizada para validações automáticas de dados e definição de modelos de dados estruturalmente tipados.
- **Dotenv**: Implementada para gestão de variáveis de ambiente de forma segura, mantendo as configurações sensíveis fora do código-fonte principal.

### Bancos de Dados

- **MongoDB**: Banco de dados NoSQL utilizado para o armazenamento de documentos e dados não estruturados, proporcionando flexibilidade na modelagem de dados.
- **Neo4j**: Banco de dados orientado a grafos, integrado para suportar operações complexas de consulta em dados estruturados como grafos, potencializando casos de uso como GraphRAG.
- **Redis**: Usado como um broker de mensagens para filas de tarefas, facilitando a execução de tarefas distribuídas e armazenamento em cache.

### Arquitetura de Contêineres

- **Docker**: Containers Docker encapsulam o aplicativo e suas dependências, fornecendo uma plataforma coesa para implantação e escalabilidade.

## Arquitetura do Sistema

A arquitetura adotada é baseada em microserviços, organizada em componentes individuais que colaboram para executarem as funcionalidades:

1. **API Principal**: Implementada com FastAPI, ela gerencia as solicitações HTTP, oferecendo endpoints para interação com os usuários e sistemas.
2. **Gerenciador de Filas**: Utiliza Redis para organização e execução de tarefas em fila, assegurando a priorização adequada das solicitações.
3. **Workers Assíncronos**: Executam a lógica de processamento de documentos e operações de IA em tarefas paralelas, maximizando a eficiência computacional.
4. **Repositórios de Dados**: Corresponsáveis pela abstração do acesso ao MongoDB e Neo4j, simplificando as operações de CRUD (Create, Read, Update, Delete) e consultas complexas.
5. **Serviços de IA**: Componentes dedicados à comunicação com APIs externas de IA, como Azure OpenAI e Google Gemini, ampliando as capacidades analíticas da aplicação.

```python
from fastapi import APIRouter, Path, Body
from typing import Literal, Optional
from app.schemas.geral import StandardResponse
from app.core.logger import Logger
from app.core.queue import QueueManager, DEFAULT_QUEUE, HIGH_PRIORITY_QUEUE, LOW_PRIORITY_QUEUE
from app.usecases.queue_usecase import get_prioridade_task_minuta, get_prioridade_taks_documentos

router = APIRouter(
    prefix="/v1",
    tags=["v1"],
    responses={404: {"description": "Not found"}},
)

logger = Logger("app.api.v1")
logger.info("Iniciando API v1.")

queue_manager = QueueManager()

@router.get("/")
def root() -> StandardResponse:
    return StandardResponse.response(message="Hello World")

@router.get("/gerar-minuta/{id}")
def gerar_minuta_endpoint(
    id: str = Path(..., description="ID único do modelo a ser gerado", example="685c6a9c890098de44ea7e0f")
) -> StandardResponse:
    """
    Gera uma minuta baseada no ID fornecido.
    
    Esta rota inicia o processamento de um pipeline de geração de minuta
    usando o ID especificado e retorna o identificador da tarefa criada.
    """
    try:
        job_id = queue_manager.enqueue_task({
            "task": "processar_pipeline_minuta_task",
            "args": (id,)
        }, get_prioridade_task_minuta(id))
        return StandardResponse.response(data={"job_id": job_id})
    except Exception as e:
        error_message = f"Erro ao criar tarefa de geração de minuta: {e}"
        logger.error(error_message)
        return StandardResponse.response(message=error_message, status_code=500)
```

### Arquivo: `app/core/logger.py`

```python
import logging
import os
from logging.handlers import RotatingFileHandler
from app.core.config import Config

LEVEL_MAP = {
    'DEBUG': logging.DEBUG,
    'INFO': logging.INFO,
    'WARNING': logging.WARNING,
    'ERROR': logging.ERROR,
    'CRITICAL': logging.CRITICAL
}

class Logger:
    def __init__(self, name: str, log_file_name: str = None):
        config = Config()
        
        if not log_file_name:
            log_file_name = "app.log"
        
        self.log_file = os.path.join(config.log_dir, log_file_name)
        self.log_level = config.log_level
        
        os.makedirs(os.path.dirname(self.log_file), exist_ok=True)
        
        self.handler = RotatingFileHandler( 
            self.log_file, 
            maxBytes=1024 * 1024 * config.log_max_mb,
            backupCount=config.log_backup_count
        )
        
        self.formatter = logging.Formatter(
            '%(asctime)s [%(levelname)s] %(name)s: %(message)s',
            datefmt='%Y-%m-%d %H:%M:%S'
        )
        self.handler.setFormatter(self.formatter)
        
        self.logger = logging.getLogger(name)
        self.logger.setLevel(LEVEL_MAP.get(self.log_level.upper(), logging.INFO))
        
        if not self.logger.handlers:
            self.logger.addHandler(self.handler)

    def debug(self, message: str):
        self.logger.debug(message)

    def info(self, message: str):
        self.logger.info(message)

    def warning(self, message: str):
        self.logger.warning(message)

    def error(self, message: str):
        self.logger.error(message)

    def critical(self, message: str):
        self.logger.critical(message)

    def exception(self, message: str):
        self.logger.exception(message)

def init_logger_ray(log_file_name: str = "ray.log"):
    config = Config()
    logger = logging.getLogger("ray")
    logger.setLevel(LEVEL_MAP.get(config.log_level.upper(), logging.INFO))
    logger.addHandler(RotatingFileHandler( 
        os.path.join(config.log_dir, log_file_name), 
        maxBytes=1024 * 1024 * config.log_max_mb,
        backupCount=config.log_backup_count
    ))
    return logger
```

### Fluxos de Dados

Os fluxos de dados são projetados cuidadosamente para maximizar eficiência e robustez:

1. **Recebimento de Requisição**: Um cliente comunica-se com a API FastAPI, disparando um pedido para realizar operações específicas como geração de minutas.
   
2. **Roteamento e Procedimento**: A requisição é roteada através dos endpoints definidos, cada uma tendo a lógica necessária para iniciar tarefas via `QueueManager`.

3. **Processamento com Ray**: Tarefas são enfileiradas e workers em segundo plano, através do Ray, executam a lógica designada paralelamente, permitindo performar computações de alta carga.

4. **Persistência e Resultado**: Ao término do processamento, resultados são salvos em bases como MongoDB ou Neo4j, ou retornados diretamente para usuários conforme apropriado, permitindo que dados críticos sejam acessíveis e reutilizáveis conforme necessidade.

5. **Ciclo de Vida de Respostas e Mensagens de Log**: Respostas são documentadas através de logs para todas as etapas do processamento, permitindo monitoramento extensivo e pós-análise efetiva.
