# Documentação Técnica Detalhada do Projeto RAG-MPMG

## Visão Geral

O projeto RAG-MPMG é uma solução de inteligência artificial voltada para o processamento e análise de documentos legais. O principal objetivo dessa aplicação é extrair informações valiosas a partir de documentos normativos internos do Ministério Público de Minas Gerais (MPMG), utilizando uma combinação de tecnologias de última geração focadas em eficiência e escalabilidade. Este documento oferece uma visão técnica aprofundada do código-fonte, detalhando a estrutura interna do projeto, as classes, funções e os fluxos de dados, assim como as tecnologias e práticas adotadas.

## Tecnologias

### Infraestrutura e Integrações

- **FastAPI**: Utilizado para a construção de APIs RESTful de alta performance, permitindo rápida prototipagem e implementação de rotas de API.
- **Ray**: Tecnologia utilizada para processamento paralelo e distribuído, garantindo a execução eficiente de tarefas assíncronas.
- **MongoDB e SQL Server**: Bancos de dados utilizados para armazenamento e recuperação de dados, com MongoDB utilizado para dados não estruturados e SQL Server para dados estruturados.
- **OpenAI e Google Gemini**: Integrações com APIs de inteligência artificial, fornecendo capacidade avançada de processamento de linguagem natural.
- **Surya OCR**: Ferramenta de reconhecimento óptico de caracteres, utilizada para converter documentos digitalizados em texto pesquisável.
- **FAISS**: Biblioteca para busca vetorial de alta performance, facilitando a recuperação de informações em bases de dados multidimensionais.
- **Docker**: Utilizado para a containerização dos serviços, garantindo um ambiente de execução consistente e isolado.

### Arquitetura

A arquitetura do projeto é modular, permitindo fácil manutenção e escalabilidade. O código está organizado em diretórios específicos, cada um com responsabilidade clara:

- **app/**: Diretório central com subdiretórios para API, lógica de negócios, acesso a dados, esquemas de dados e serviços auxiliares.
- **RAG/**: Pontual para o pipeline de recuperação, análise e geração de documentos.
- **static/**: Armazena arquivos estáticos utilizados na aplicação.
- **main.py**: Ponto de entrada da aplicação, onde a configuração e inicialização dos serviços ocorrem.
- **requirements.txt**: Lista de dependências do projeto.
- **README.md**: Documento fornecendo instruções de instalação e execução do projeto.

### docker-compose.yml

O arquivo `docker-compose.yml` é vital para a orquestração dos serviços, definindo contêineres, redes e volumes. Este arquivo é dividido em diferentes serviços, cada um configurado para um propósito específico.

#### Serviço Qdrant

O Qdrant é utilizado para armazenamento e consulta de dados vetoriais. O contêiner é configurado com a imagem do Qdrant mais recente. Volumes são especificados para garantir a persistência dos dados:

- **Volumes**: Configurado em `./qdrant_data`, assegurando que os dados sejam preservados entre reinicializações.
- **Healthcheck**: Inclui verificações de saúde que monitoram o status do serviço a cada 30 segundos.

#### Serviço Neo4j

Neo4j é utilizado para a organização de dados em um grafo. O contêiner é configurado com a versão comunitária do Neo4j. Vários volumes são configurados para logs, dados e plugins, assegurando que todas as informações relevantes sejam armazenadas de forma persistente.

- **Volumes**: Includem `logs`, `import`, `plugins`, e um volume para dados principal.
- **Healthcheck**: Implementado usando `wget`, garantindo que o endpoint esteja respondendo corretamente.

### main.py

O `main.py` serve como ponto principal de configuração e execução da API FastAPI. As variáveis de ambiente são carregadas através do `dotenv`, e o aplicativo FastAPI é inicializado com propriedades configuradas de acordo com `Config`.

#### Estrutura do Código

```python
from dotenv import load_dotenv
import ray
import fastapi
from app.api.v1 import router as v1_router
from app.core.config import Config

load_dotenv()
config = Config()

app = fastapi.FastAPI(
    title="MP-IA",
    version=config.app_version,
    description="API - para processamento de Inteligência Artificial",
    openapi_url="/openapi.json",
    docs_url="/docs",
    redoc_url="/redoc",
)

app.include_router(v1_router)
```

Este trecho de código define o ponto de entrada da aplicação, carrega as configurações necessárias e inicia o servidor FastAPI, que serve todas as rotas configuradas.

### app/api/v1.py

Este arquivo implementa as rotas da API que servem como interface para a interação com o sistema. Cada endpoint é responsável pela execução de tarefas específicas, desde geração de documentos a execução de operações de OCR.

#### Detalhes dos Endpoints

1. **root()**: Fornece um endpoint base simples que retorna "Hello World".
2. **gerar_minuta_endpoint()**: Endpoint que inicia um pipeline para gerar um modelo de minuta com base em um ID de entrada.
3. **modificar_minuta_endpoint()**: Modifica um modelo de minuta existente, permitindo ajustes e atualizações.
4. **ocr_documentos_processo_endpoint()**: Inicia uma operação OCR em documentos associados a um processo específico.
5. **get_job_status()**: Retorna o status de um job em execução, permitindo monitoramento de processos assíncronos.
6. **get_queue_info()**: Proporciona visualização das estatísticas gerais da fila de processamento.

Os handlers de erros são implementados para capturar exceções e assegurar que respostas apropriadas sejam enviadas ao cliente.

### app/core/config.py

A configuração central é definida na classe `Config`, que encapsula todas as variáveis de ambiente necessárias para a operação da aplicação. Ela é chave para a configuração do acesso a serviços externos e recursos internos.

#### Principais Componentes:

- **Variáveis de Ambiente**: Gerencia as strings de conexão ao MongoDB, SQL Server e o ambiente do Ray.
- **Métodos**: `.get_ray_runtime_env()` facilita a configuração do ambiente para execução do Ray.

### app/core/logger.py

O módulo de logging fornece uma infraestrutura robusta para captura e gerenciamento de logs. A classe `Logger` utiliza `RotatingFileHandler` para criar logs rotativos, garantindo que os arquivos de log não cresçam excessivamente.

#### Funções Importantes:

- **Configuração do Logger**: Inicialização dos handlers e formatos de log.
- **Níveis de Log**: Métodos para `debug`, `info`, `warning`, `error` e `critical`, com suporte completo a stack traces.

### app/core/queue.py

Gerenciamento de filas é realizado pelo `QueueManager`, que utiliza Redis para coordenação de jobs. Essa implementação assegura a organização e priorização de tarefas assíncronas no sistema.

#### Funcionalidades:

- **Enqueue e Dequeue**: Adiciona e remove tarefas da fila, garantindo execução ordenada.
- **Consulta de Status**: Permite verificação do progresso e estado das tarefas na fila.

### app/services/ocr_service.py

O serviço de OCR é responsável por processar documentos em diferentes formatos, utilizando Surya OCR para converter imagens em texto pesquisável. Isto é crítico para transformar documentos escaneados em dados manipuláveis.

### app/usecases/inteligencia_usecase.py

Este componente gerencia casos de uso específicos relacionados à manipulação de documentos. Fornece a lógica para operações como pesquisa e salvamento de dados extraídos de documentos.

### Fluxos de Dados

O fluxo de dados começa com a recepção de documentos que são então enviados pelo pipeline de processamento de texto, aplicado a operações de OCR e finalmente armazenados em bancos de dados apropriados. O sistema é desenhado para suportar alto volume, assegurando que operações de consulta e recuperação sejam rápidas e eficientes.
