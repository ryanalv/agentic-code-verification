### O que é este sistema?
Imagine um assistente jurídico virtual extremamente eficiente que trabalha para o Ministério Público de Minas Gerais (MPMG). Este sistema, chamado **MP-IA**, é capaz de ler milhares de documentos processuais (como PDFs, HTMLs e DOCs), extrair seu conteúdo textual, "entender" o contexto jurídico e redigir automaticamente peças processuais complexas como denúncias, arquivamentos e representações.

### Analogia Simples
Pense no sistema como uma **fábrica de documentos jurídicos inteligente**:
- **Matéria-prima:** Documentos digitais de processos (fotos de processos, PDFs, etc.)
- **Operários (Workers):** Computadores especializados em ler (OCR) e escrever (IA)
- **Esteira de produção:** Filas organizadas por prioridade (urgente, normal, baixa)
- **Produto final:** Minutas jurídicas prontas para revisão humana

### Por que isso é importante?
O sistema reduz de horas para minutos o tempo de preparação de documentos jurídicos, permitindo que promotores e analistas foquem em análise estratégica em vez de digitação e formatação repetitiva.

## 2. CONTEXTO TEÓRICO DA ARQUITETURA

Antes de mergulharmos no código, é essencial compreender os padrões arquiteturais e tecnologias que sustentam este sistema. Esta seção explica os conceitos teóricos para depois mostrar como eles se manifestam concretamente no código.

### 2.1 RAG (Retrieval-Augmented Generation)

**Conceito Teórico:**
RAG é uma arquitetura de Inteligência Artificial que combina dois mundos:
1. **Recuperação de Informação (Retrieval):** Buscar dados relevantes de uma base de conhecimento
2. **Geração de Texto (Generation):** Usar modelos de linguagem (LLMs) para criar texto coerente baseado nos dados recuperados

**Analogia:** Imagine um advogado que, antes de escrever uma petição, consulta uma biblioteca gigante (banco de dados vetorial) para encontrar jurisprudências e leis relevantes, e só então redige o documento usando essas referências.

**Aplicação no Código:**
No arquivo `app/services/gpt_service.py`, vemos a implementação concreta:
- **Recuperação:** Os documentos OCR (`_get_prompt_documentos`) são recuperados do MongoDB
- **Geração:** O `AzureOpenAIClient` gera texto baseado nesses documentos + templates jurídicos (`get_prompt_base`)

### 2.2 Arquitetura de Filas Distribuídas (Queue-Based Architecture)

**Conceito Teórico:**
Em sistemas de alta carga, processar tudo imediatamente pode sobrecarregar servidores. A solução é criar "listas de espera inteligentes" (filas) onde tarefas aguardam ser processadas conforme recursos ficam disponíveis.

**Analogia:** Como uma fila de banco organizada por prioridade (idosos, gestantes primeiro), o sistema separa trabalhos pesados (muitos documentos) dos leves.

**Aplicação no Código:**
O arquivo `app/core/queue.py` implementa o `QueueManager` usando Redis:
- Filas: `high` (prioridade), `default` (normal), `low` (pesada)
- Status tracking: `pending` → `processing` → `completed`
- Mecanismo BLPOP (Blocking Left Pop) para workers aguardarem novas tarefas

### 2.3 Computação Distribuída com Ray

**Conceito Teórico:**
Ray é um framework que permite executar Python em paralelo em múltiplas máquinas, como se fosse uma única máquina gigante. Ele gerencia automaticamente a alocação de CPUs, GPUs e memória.

**Analogia:** Em vez de um único funcionário fazer 100 cópias de um documento (sequencial), Ray contrata 100 funcionários temporários (workers remotos) que trabalham simultaneamente.

**Aplicação no Código:**
Em `app/workers/inteligencia_worker.py`:
- Decoradores `@ray.remote(num_gpus=1, num_cpus=2)` definem recursos necessários
- `processar_pipeline_minuta_task` orquestra tarefas paralelas de OCR
- `ray.get()` agrega resultados de múltiplas máquinas

### 2.4 Bancos de Dados Poliglota (Polyglot Persistence)

**Conceito Teórico:**
Usar diferentes tipos de bancos de dados para diferentes finalidades:
- **Documentos (MongoDB):** Dados estruturados flexíveis (JSON)
- **Cache/Filas (Redis):** Acesso ultra-rápido, temporário
- **Relacional (SQL Server):** Dados transacionais legados (PJe/SRU)

**Aplicação no Código:**
- `app/core/db/mongodb.py`: Persistência de minutas e documentos
- `app/core/db/redis.py`: Locks distribuídos e filas
- `app/repositories/inteligencia_repository.py`: Abstração sobre SQL Server e HDFS

### 2.5 Estratégia de Fallback e Resiliência

**Conceito Teórico:**
Sistemas distribuídos falham. A estratégia de fallback garante que se um serviço (Azure OpenAI) falhar, outro (Google Gemini) assuma automaticamente.

**Analogia:** Como um carro com pneu step - se um furar, você continua a viagem, embora mais devagar.

**Aplicação no Código:**
Em `app/services/gpt_service.py`:
- Lock Redis (`lock_gemini`) ativa modo de contingência
- Exceções `ACQUIRE_LOCK_EXCEPTIONS` (RateLimitError, etc.) disparam fallback
- Alternância entre `AzureOpenAIClient` e `GoogleGeminiClient`

```python
@router.get("/gerar-minuta/{id}")
def gerar_minuta_endpoint(
    id: str = Path(..., description="ID único do modelo a ser gerado", 
                  example="685c6a9c890098de44ea7e0f")
) -> StandardResponse:
    try:
        # 1. Determina prioridade baseada na quantidade de documentos
        fila = get_prioridade_task_minuta(id)
        
        # 2. Prepara payload da task
        task_payload = {
            "task": "processar_pipeline_minuta_task",
            "args": (id,)  # Tupla de argumentos para a função Ray
        }
        
        # 3. Enfileira no Redis com prioridade calculada
        job_id = queue_manager.enqueue_task(task_payload, queue=fila)
        
        return StandardResponse.response(
            message="Minuta enfileirada para geração",
            data={"job_id": job_id, "fila": fila}
        )
    except Exception as e:
        logger.error(f"Erro ao enfileirar minuta: {str(e)}")
        return StandardResponse.response(
            message="Erro ao processar solicitação",
            status_code=500
        )
```

**Lógica de Negócio Importante:**
A priorização é dinâmica: o sistema consulta o documento de inteligência no MongoDB, conta quantos documentos precisam ser processados (`len(documento['documentos'])`), e decide:
- **HIGH_PRIORITY**: Poucos documentos (processamento rápido)
- **DEFAULT**: Quantidade média
- **LOW_PRIORITY**: Muitos documentos (evita bloquear fila para outros usuários)

def get_ray_runtime_env(self) -> RuntimeEnv:
    with open("requirements.txt", "r") as f:
        pip_requirements = [line.strip() for line in f 
                           if line.strip() and not line.startswith("#")]
    
    return RuntimeEnv(
        pip=pip_requirements,
        config=RuntimeEnvConfig(setup_timeout_seconds=1800),
        working_dir=".",
        env_vars={
            "MONGO_URI": self.mongo_uri,
            "AZURE_OPENAI_API_KEY": self.azure_openai_api_key,
            # ... todas as variáveis sensíveis são injetadas aqui
        }
    )

3. O timeout de 1800s permite instalação de pacotes pesados (PyTorch, etc.)

#### 4.2.3 Gerenciamento de Filas (`app/core/queue.py`)

O `QueueManager` é o coração da orquestração assíncrona. Ele abstrai o Redis como uma fila de prioridade robusta.

**Estrutura de Dados no Redis:**

```
fila_ray:high          → Lista LPUSH/RPOP (alta prioridade)
fila_ray:default       → Lista LPUSH/RPOP (prioridade normal)
fila_ray:low           → Lista LPUSH/RPOP (baixa prioridade)
fila_ray:job:{id}:status → Hash (metadados do job)
```

**Métodos Principais:**

1. **Enqueue (Enfileirar):**
```python
def enqueue_task(self, data: Dict[str, Any], queue: str = DEFAULT_QUEUE) -> str:
    job_id = data.get("job_id", str(uuid4()))
    data["job_id"] = job_id
    
    # Atualiza status para pending com TTL de 24h
    self._set_job_status(job_id, STATUS_PENDING, queue, data_raw=data)
    
    # Adiciona à fila específica (RPUSH = adiciona ao final)
    queue_name = self._get_queue_name(queue)
    self.redis_client.rpush(queue_name, json.dumps(data))
    
    return job_id
```

2. **Dequeue (Desenfileirar):**
```python
def dequeue_task(self, queues: Optional[List[str]] = None, 
                 block: bool = True, timeout: int = 5):
    queue_names = [self._get_queue_name(q) for q in (queues or [HIGH_PRIORITY_QUEUE, DEFAULT_QUEUE, LOW_PRIORITY_QUEUE])]
    
    if block:
        # BLPOP: Blocking Left Pop - aguarda até ter item em qualquer fila
        # Prioridade respeita ordem da lista (high > default > low)
        result = self.redis_client.blpop(queue_names, timeout=timeout)
        if result:
            queue_name, job_data = result
            job_data = json.loads(job_data)
            self._set_job_status(job_data["job_id"], STATUS_PROCESSING, queue_name)
            return queue_name, job_data
    else:
        # Modo não-bloqueante (polling)
        for queue_name in queue_names:
            job_data = self.redis_client.lpop(queue_name)
            if job_data:
                job_data = json.loads(job_data)
                self._set_job_status(job_data["job_id"], STATUS_PROCESSING, queue_name)
                return queue_name, job_data
    return None
```

**Recuperação de Falhas:**
O método `requeue_processing_jobs()` é crucial para resiliência:
```python
def requeue_processing_jobs(self):
    """Recoloca jobs 'processing' na fila após crash do worker"""
    pattern = f"{QUEUE_PREFIX}:job:*:status"
    for key in self.redis_client.scan_iter(match=pattern):
        status_data = json.loads(self.redis_client.get(key))
        if status_data.get("status") == STATUS_PROCESSING:
            job_data = status_data.get("data_raw")
            queue = status_data.get("queue")
            self.enqueue_task(job_data, queue)
            self._set_job_status(job_data["job_id"], STATUS_PENDING, queue, data_raw=job_data)
```

- `avg_wait_time`: Tempo na fila (enqueue → processing)
- `avg_process_time`: Tempo de execução (processing → completed)
- `avg_time_elapsed`: Tempo total (enqueue → completed)

**Builder Pattern para Prompts:**
A classe `PromptBuilder` permite composição incremental:
```python
class PromptBuilder:
    def __init__(self, base=None):
        self.partes = []
        if base:
            self.add(base)
    
    def add(self, texto):
        self.partes.append(str(texto))
        return self  # Fluent interface
    
    def add_line_break(self, qtd=1):
        self.partes.append("\n" * qtd)
        return self
    
    def build(self, separador=" "):
        # Lógica inteligente: não duplica quebras de linha
        resultado = []
        for i, parte in enumerate(self.partes):
            if i > 0 and not self.partes[i-1].endswith('\n') and not parte.startswith('\n'):
                resultado.append(separador)
            resultado.append(parte)
        return "".join(resultado).strip()
```

**Exemplo de Uso:**

    .add_line_break(2)
    .add("Analise o seguinte caso:")
    .add_line_break()
    .add(descricao_caso)
    .build("\n")

def preparar_tasks_ray(documentos_processar, 
                      obter_texto_pdf_task, 
                      obter_texto_documento_task):

            # Modelos sempre processados como documentos genéricos

        else:

                minutas_encontradas.append(doc)
            else:
                # Decide qual worker usar baseado na extensão
                extensao = get_extensao_documento_pje(doc.id) if doc.tipo == 'sru' else ...
                if extensao == '.pdf':
                    remote_tasks.append(obter_texto_pdf_task.remote(doc))
                else:
                    remote_tasks.append(obter_texto_documento_task.remote(doc))
    
    return minutas_encontradas, remote_tasks

**Validação de Qualidade:**
Após processamento paralelo, valida taxa de sucesso:
```python
def validar_ocrs(ocrs: list[DocumentoInteligenciaDTO]):
    ocrs_validos = [o for o in ocrs if o is not None]
    total = len(ocrs)
    
    if not ocrs_validos:
        raise Exception("Nenhum OCR foi gerado com sucesso")
    
    taxa_falha = (total - len(ocrs_validos)) / total
    if taxa_falha > 0.1:  # Mais de 10% de falha
        raise Exception(f"Taxa de falha no OCR muito alta: {taxa_falha:.2%}")
    
    return ocrs_validos
```

```python
def processar_resposta_ray(documento, funcao, ray_context):
    try:
        resultado = funcao(documento)
        if resultado is None:
            logger.warning(f"OCR retornou None para {documento.id}")
        return resultado
    except Exception as e:
        # Monta mensagem detalhada com traceback
        err = f"Erro: {str(e)}\nTraceback: {traceback.format_exc()}"
        
        # Persiste o erro no MongoDB para análise posterior
        update = {
            "warnings": {
                "id": documento.id,
                "id_task": ray_context.get_task_id(),
                "warning": err
            }
        }
        atualizar_documento_inteligencia(documento.id_documento, update, push=True)
        return None
```

**Padrão Dead Letter Queue implícito:** Erros são registrados no documento pai, permitindo retry manual ou análise posterior sem perder o contexto.

## 7. TRATAMENTO DE ERROS E RESILIÊNCIA

### 7.1 Estratégias Implementadas

| Estratégia | Implementação | Objetivo |
|------------|--------------|----------|
| **Retry com Fallback** | `gpt_service.py` - tenta Azure, cai para Gemini | Disponibilidade 99.9% |
| **Circuit Breaker** | Lock Redis (`lock_gemini`) por 60s após falha | Evitar retry storm |
| **Dead Letter Queue** | `processar_resposta_ray` - salva erro no MongoDB | Debugging e retry manual |
| **Requeue Automático** | `requeue_processing_jobs()` - resgata jobs "processing" | Recuperação após crash |
| **Validação de Qualidade** | `validar_ocrs()` - rejeita se >10% falha | Garantir qualidade do output |
| **TTL em Cache** | Redis keys com `ex=60` | Evitar dados obsoletos |

### 7.2 Erro de Conexão RAG (Contexto Específico)

O erro mencionado no contexto (`ConnectionResetError(10054)`) indica problemas de conectividade em ambientes distribuídos. No código, isso é mitigado por:

1. **Timeouts configuráveis** no `RuntimeEnvConfig(setup_timeout_seconds=1800)`
2. **Exceções específicas** capturadas: `RemoteProtocolError`, `ReadTimeout`
3. **Graceful degradation**: Se Ray falhar, o sistema pode reenfileirar jobs

## 8. CONSIDERAÇÕES DE DEPLOY E OPERAÇÃO

### 8.1 Requisitos de Infraestrutura

| Componente | Requisito | Justificativa |
|------------|-----------|---------------|
| **Redis** | Alta disponibilidade (Sentinel ou Cluster) | Fila é crítica; perda de dados = perda de jobs |
| **MongoDB** | Replica Set (3 nós) | Dados de inteligência não podem ser perdidos |
| **Ray Cluster** | 1+ nós com GPU (NVIDIA) | OCR de PDFs requer processamento GPU |
| **SQL Server** | Acesso via rede interna (VPN) | Dados sensíveis do PJe |
| **HDFS** | Conectividade via protocolo Hadoop | Documentos MPE legados |

### 8.2 Variáveis de Ambiente Críticas

```bash
# IA e APIs

AZURE_OPENAI_API_KEY=***
AZURE_OPENAI_DEPLOYMENT=gpt-4-turbo

### 8.3 Monitoramento Sugerido

1. **Métricas Redis:** Tamanho das filas (`LLEN`), taxa de processamento
2. **Métricas Ray:** Utilização de GPU, tempo médio de tasks, falhas
3. **Métricas Aplicação:** Jobs pendentes, taxa de fallback para Gemini, erros de OCR
4. **Logs:** Centralizar em ELK ou similar; atenção especial aos `scheduler.log`

## 9. GLOSSÁRIO

| Termo | Definição |
|-------|-----------|
| **OCR** | Optical Character Recognition - tecnologia para extrair texto de imagens/PDFs |
| **RAG** | Retrieval-Augmented Generation - arquitetura que combina busca de documentos + geração de texto |
| **Minuta** | Documento jurídico pré-formatado (denúncia, arquivamento, etc.) |
| **Vector Store** | Banco de dados otimizado para busca semântica (similaridade de significado, não apenas palavras-chave) |
| **Ray** | Framework de computação distribuída para Python |
| **DTO** | Data Transfer Object - objeto simples para transferência de dados entre camadas |
| **Upsert** | Operação que insere se não existe, ou atualiza se existe (Update + Insert) |
| **BLPOP** | Blocking Left Pop - comando Redis que aguarda até haver item na fila |
| **Fallback** | Estratégia de contingência quando serviço primário falha |
